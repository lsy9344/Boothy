# Story 1.2: 이름+뒤4자리 기반 세션 시작과 내구적 세션 생성

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

booth customer로서,
이름과 휴대전화 뒤 4자리만으로 세션을 시작하고 싶다.
그래서 예약 확인이나 전체 전화번호 입력 없이 빠르게 부스에 들어갈 수 있다.

## Acceptance Criteria

1. 부스 시작 화면에서 고객이 비어 있지 않은 이름과 유효한 네 자리 숫자 휴대전화 뒤4자리를 입력하면, 시스템은 고객용 `boothAlias`와 별도의 내부 `sessionId`를 가진 활성 세션을 생성해야 한다. 또한 세션은 세션 범위 파일시스템 루트와 초기 `session.json` 매니페스트에 영속화되어야 한다.
2. 부스 시작 화면에서 고객이 빈 이름, 숫자가 아닌 suffix, 또는 네 자리가 아닌 suffix를 입력하면, 시스템은 다음 단계로 진행을 막고 plain-language 검증 안내를 보여줘야 한다. 또한 전체 전화번호 입력이나 예약 검증을 요구하면 안 된다.

## Dev Notes

### 스토리 범위와 선행조건

- 이 스토리는 "폼 하나 추가"가 아니라, 고객 입력을 host-owned durable session truth로 연결하는 첫 세션 라이프사이클 진입점이다.
- Story 1.1이 아직 `ready-for-dev` 상태이고 현재 워크스페이스에도 실제 앱 scaffold가 없다. 따라서 Story 1.2 구현은 Story 1.1 완료 또는 동등한 bootstrap 결과물이 선행되어야 한다.
- 이 스토리의 범위는 세션 시작과 초기 세션 영속화까지다. 프리셋 카탈로그, 카메라 준비 상태, 촬영, 삭제, 타이밍, `Preview Waiting`, `Phone Required` 완성은 후속 스토리 범위다.

### 현재 워크스페이스 상태

- 현재 저장소에는 planning artifacts와 Story 1.1 문서만 있고, `package.json`, `src/`, `src-tauri/`, `Cargo.toml` 같은 실제 실행 코드 골격은 아직 없다.
- 따라서 아래 파일 경로는 Story 1.1이 만든 구조를 전제로 한 "목표 경로"다. Story 1.1이 같은 경계를 다른 파일명으로 구현했다면 경계는 유지하고 경로만 합리적으로 조정해도 된다.
- Git 저장소가 초기화되어 있지 않아 commit history 기반 학습은 사용할 수 없다.

### 이전 스토리 인텔리전스

- Story 1.1은 공식 `Vite react-ts + manual Tauri CLI initialization`을 고정했고, `/booth`, `/operator`, `/authoring`, `/settings` 최상위 surface와 typed adapter seam을 선행 조건으로 정의했다.
- Story 1.2는 그 구조를 재사용해야 하며, 세션 시작을 위해 새 top-level route나 direct `invoke` 패턴을 만들면 안 된다.
- Story 1.1은 `shared-contracts`, `session-domain`, `src-tauri/src/commands/`, `src-tauri/src/session/` 같은 구조를 미리 열어 두라고 요구했다. Story 1.2는 바로 그 seam 위에 구현되어야 한다.

### 핵심 구현 목표

- 고객 입력은 정확히 두 개다: `name`, `phoneLastFour`.
- 고객이 보는 식별자는 `boothAlias`이고, 저장소/계약/상태 상관관계의 durable identity는 opaque `sessionId`다.
- 세션의 durable truth는 세션별 파일시스템 루트와 `session.json`이 소유한다.
- React는 host가 정규화해 돌려준 결과만 소비하고, durable truth를 직접 만들거나 수정하지 않는다.

### 아키텍처 준수사항

- 활성 세션의 진실은 route state, component state, SQLite가 아니라 session-scoped filesystem root가 소유해야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Core Architectural Decisions]
- 고객용 `boothAlias`는 이름+뒤4자리 조합을 유지하되 durable `sessionId`와 분리되어야 한다. [Source: _bmad-output/planning-artifacts/epics.md#Additional Requirements]
- Rust host만 세션 파일을 생성/수정할 수 있다. React나 plugin-fs가 durable 세션 생성의 권위 경로가 되면 안 된다. [Source: _bmad-output/planning-artifacts/architecture.md#Authentication & Security]
- 프론트엔드와 호스트 사이 계약은 TypeScript의 `Zod 4` 검증과 Rust 재검증을 함께 사용해야 한다. [Source: _bmad-output/planning-artifacts/epics.md#Additional Requirements]
- React Router는 top-level surface 진입에만 쓰고, 실제 세션 워크플로 truth는 state-driven으로 유지해야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Frontend Architecture]
- Tauri commands는 request/response 경계이고, event/channels는 이후 상태 스트림에 사용한다. Story 1.2에서는 `start_session` command가 기본 진입점이다. [Source: _bmad-output/planning-artifacts/architecture.md#API & Communication Patterns]

### 세션 매니페스트 계약 가이드

- 이 스토리에서 `session.json` 계약을 최소 v1로 동결한다. Story 4-6용 frozen contract baseline과 충돌하지 않도록 `schemaVersion`을 명시한다.
- v1 최소 필드 권장안:
  - `schemaVersion`
  - `sessionId`
  - `boothAlias` 또는 동등한 고객용 alias 구조
  - `createdAt` / `updatedAt`
  - 초기 lifecycle/status 필드
  - `activePreset` 또는 동등한 nullable preset binding placeholder
  - `captures` 빈 배열
  - `postEnd` nullable placeholder
- 이 필드 집합은 현재 story 구현에 필요한 최소치를 우선하고, 후속 story가 사용할 render/capture correlation 확장 필드를 수용할 수 있어야 한다.
- Story 1.1이나 별도 계약 문서가 더 풍부한 스키마 뼈대를 이미 제공했다면, 그 뼈대를 확장하고 병렬 manifest shape를 새로 만들지 말 것.

### UX / 카피 가드레일

- 고객 시작 단계는 `이름 + 휴대전화 뒤4자리` 입력만으로 끝나야 한다. 전체 전화번호, 예약번호, 운영자 확인 단계를 넣지 말 것. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Reading Guide]
- 입력 검증은 즉시 plain-language로 안내해야 하며, 고객이 잘못된 형식의 이유를 이해할 수 있어야 한다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Form Patterns]
- 고객 문구는 낮은 문구 밀도 원칙을 지켜야 하며 기술 진단어를 포함하면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#NFR-001 Customer Guidance Density and Simplicity]
- 세션 시작부터 부스 별칭과 이후 타이밍 정보를 상시 인지 가능하게 둘 예정이므로, 이번 story에서도 성공 후 alias를 쉽게 재사용 가능한 state에 올려 두는 편이 좋다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Journey Patterns]

### 기술 요구사항 및 최신 공식 문서 확인

- 2026-03-20 기준 Tauri 공식 release 페이지에는 `tauri` `v2.10.3`, `@tauri-apps/api` `v2.10.1`, `tauri-cli` `v2.10.1`, `@tauri-apps/cli` `v2.10.1`이 표시된다. Story 1.1이 이미 호환 가능한 Tauri v2 조합을 고정했다면, Story 1.2는 그 v2 command/plugin 모델을 따르고 불필요한 버전 churn은 피한다. [Source: https://v2.tauri.app/release/]
- Tauri 공식 문서는 프론트엔드가 `@tauri-apps/api/core`의 `invoke`로 Rust `#[tauri::command]` 함수를 호출하는 패턴을 안내한다. 따라서 `start_session`은 Rust command, `startSession()`은 TypeScript adapter wrapper로 두는 것이 현재 아키텍처와 일치한다. [Source: https://v2.tauri.app/develop/calling-rust/]
- Tauri 공식 fs 문서는 Rust 쪽 파일 조작에는 `std::fs` 또는 `tokio::fs` 사용을 안내한다. 이는 "host만 세션 파일을 mutate한다"는 아키텍처와 일치한다. [Source: https://v2.tauri.app/plugin/file-system/]
- 같은 fs 문서는 경로를 base directory 또는 path API 기준으로 다루고 path traversal을 막는다. 구현 판단상 durable 세션 루트는 bundled `Resources`가 아니라 host가 해석한 app data/app local data 계열 경로 아래 두는 편이 안전하다. 이것은 Tauri fs 문서의 base directory/security 설명과 본 프로젝트 아키텍처를 결합한 추론이다. [Source: https://v2.tauri.app/plugin/file-system/]
- Zod 공식 문서는 2026-03-20 현재 `Zod 4`를 stable로 안내한다. Story 1.2에서는 booth start input, session manifest DTO, command payload 검증의 기준선으로 사용한다. [Source: https://zod.dev/]

### 테스트 요구사항

- 폼 단위
  - 빈 이름, 공백-only 이름, 4자리가 아닌 suffix, 숫자가 아닌 suffix가 모두 차단되는지 검증
  - 유효 입력에서 adapter가 호출되는지 검증
- 계약 단위
  - `SessionStartInput` / `SessionStartResult` Zod schema 테스트
  - `session.json` v1 parse/serialize 테스트
- Rust 단위
  - `start_session`가 session root와 초기 `session.json`을 생성하는지 검증
  - 중간 실패 시 partial manifest 또는 잘못된 디렉터리 잔여물이 남지 않는지 검증
- 범위 제외
  - preset selection, camera readiness, capture persistence, preview waiting, deletion policy, timing alerts는 이 story 테스트 범위가 아니다

### 금지사항 / 안티패턴

- React screen/component에서 직접 `invoke('start_session')` 호출 금지
- session truth를 route params, localStorage, SQLite에 먼저 저장하고 나중에 파일시스템으로 옮기는 임시 설계 금지
- 전체 전화번호, 예약 ID, 운영자 승인 등 추가 입력 요구 금지
- customer-facing UI에 `sessionId`, 파일 경로, manifest, darktable, XMP, helper, sidecar 같은 내부 용어 노출 금지
- Story 1.2에서 preset 선택, 타이머, capture flow까지 한 번에 과구현하는 것 금지
