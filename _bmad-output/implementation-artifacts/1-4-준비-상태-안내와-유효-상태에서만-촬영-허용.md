# Story 1.4: 준비 상태 안내와 유효 상태에서만 촬영 허용

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

booth customer로서,
부스가 지금 촬영 가능한지 잠시 기다려야 하는지 분명히 알고 싶다.
그래서 기기 내부를 이해하지 않아도 촬영 흐름을 믿고 행동할 수 있다.

## Acceptance Criteria

1. 활성 세션과 선택된 프리셋이 있는 상태에서 카메라 또는 호스트의 readiness가 바뀌면, 부스는 런타임 truth를 `Preparing`, `Ready`, 또는 wait/call guidance 같은 plain-language 고객 상태로 번역해야 한다. 또한 고객용 문구는 승인된 low-density guidance rule 안에 머물러야 한다.
2. 부스가 승인된 capture 상태가 아닐 때 고객이 촬영을 시도하면, 촬영은 차단되어야 한다. 또한 부스는 기술 진단어를 노출하지 않고 고객에게 기다려야 하는지 도움을 요청해야 하는지 안내해야 한다.

## Dev Notes

### 스토리 목적

- 이 스토리의 핵심은 "고객이 언제 찍을 수 있는지 신뢰할 수 있게 말해주고, 안 되는 상태에서는 안전하게 막는 것"이다.
- 실제 사진 저장 성공/프리뷰 준비 분리는 Story 1.5의 책임이므로, 여기서는 readiness truth와 capture eligibility truth를 먼저 고정한다.
- 고객은 내부 장비 상태를 해석하면 안 되고, 부스는 고객 대신 plain-language 상태를 번역해 주어야 한다.

### 선행 조건과 교차 스토리 문맥

- Story 1.4는 Epic 1 Story 1.2의 활성 세션 생성과 Story 1.3의 활성 프리셋 선택을 전제로 한다. 둘 중 하나라도 없으면 capture 가능 상태가 될 수 없다. [Source: _bmad-output/planning-artifacts/epics.md#Story 1.2: 이름+뒤4자리 기반 세션 시작과 내구적 세션 생성] [Source: _bmad-output/planning-artifacts/epics.md#Story 1.3: 승인된 프리셋 카탈로그 표시와 활성 프리셋 선택]
- Story 1.1은 아직 부트스트랩 가이드 문서만 존재하며 실제 앱 스캐폴드가 확인되지 않았다. 따라서 dev agent는 Story 1.1의 구조/경계 가드레일을 따라 Story 1.4 구현 위치를 맞춰야 한다.
- 현재 implementation-artifacts에는 Story 1.1만 존재하고 Story 1.2, 1.3 story 파일은 아직 생성되지 않았다. 따라서 이번 story 문서는 의존성을 명시하되, 실제 구현 시에는 동등한 seam 또는 선행 구현 여부를 먼저 확인해야 한다.

### 제품/상태 모델 가드레일

- 고객 상태 vocabulary는 PRD의 lifecycle/state model과 충돌하면 안 된다. 최소한 `Preparing`, `Ready`, `Preview Waiting`, `Export Waiting`, `Completed`, `Phone Required`, `Warning`과의 호환성을 고려한 상태 모델이어야 한다. [Source: _bmad-output/planning-artifacts/prd.md#Core Product State Model]
- Story 1.4의 직접 목표는 readiness/capture guard지만, 이후 Story 2.4에서 `Warning` 상태에서도 촬영이 계속 가능할 수 있으므로 `displayed state`와 `capture eligibility`를 같은 boolean으로 단순화하지 말아야 한다. 이것은 PRD 상태표와 Epic 2.4 요구사항을 함께 본 해석이다. [Source: _bmad-output/planning-artifacts/prd.md#Core Product State Model] [Source: _bmad-output/planning-artifacts/epics.md#Story 2.4: 조정된 종료 시각 표시와 경고/종료 알림]
- `Phone Required`는 단순 에러 덤프가 아니라 고객 보호 상태다. readiness가 막힌 이유를 내부 코드로 보여주는 대신 "기다리기" 또는 "도움 요청" 중 고객이 해야 할 다음 행동만 남겨야 한다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Phone Required 보호 흐름]

### UX / copy 요구사항

- 고객용 문구는 plain-language와 low-density 원칙을 지켜야 하고 기술 진단어 또는 내부 운영 용어를 포함하면 안 된다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Reading Guide]
- FR-003과 NFR-001에 따라 readiness 화면은 plain-language readiness guidance를 보여주고, copy budget은 동적 값 제외 `주 문장 1개 + 보조 문장 1개 + 주요 액션 1개`를 넘기지 않아야 한다. [Source: _bmad-output/planning-artifacts/prd.md#FR-003 Readiness Guidance and Valid-State Capture] [Source: _bmad-output/planning-artifacts/prd.md#NFR-001 Customer Guidance Density and Simplicity]
- 고객에게는 "왜 카메라가 안 되는지"보다 "지금 찍을 수 있는지 / 잠시 기다릴지 / 도움을 요청할지"가 더 중요하다. copy selector는 이 정보 구조를 강제해야 한다.

### 아키텍처 준수사항

- Rust host는 세션, 타이밍, 캡처, 렌더, 완료 truth를 정규화하는 단일 계층이어야 하므로 readiness truth도 host에서 계산하고 React는 그 결과만 소비해야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#System Overview] [Source: _bmad-output/planning-artifacts/epics.md#Additional Requirements]
- React Router는 top-level surface 분리용으로만 사용하고 workflow truth는 상태 기반이어야 한다. readiness를 route 이름이나 local-only UI flag로 소유하지 말 것. [Source: _bmad-output/planning-artifacts/architecture.md#Core Architectural Decisions]
- React component는 직접 Tauri를 호출하지 않고 typed adapter/service를 통해서만 host와 통신해야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Core Architectural Decisions] [Source: _bmad-output/planning-artifacts/architecture.md#Communication & Integration Patterns]
- customer-safe text translation은 중앙 selector에 모아야 하며, architecture는 `booth-shell/selectors/customerStatusCopy.ts`를 좋은 예시로 직접 제시한다. [Source: _bmad-output/planning-artifacts/architecture.md#Pattern Examples]
- FR-003/FR-004 구현 경계는 `src/booth-shell/`, `src/capture-adapter/`, `src-tauri/src/capture/`가 중심이다. [Source: _bmad-output/planning-artifacts/architecture.md#Requirements to Structure Mapping]

### 테스트 요구사항

- readiness 상태 번역은 UI snapshot보다 selector/unit test 비중을 높여야 한다. 고객 문구 위계와 capture 가능 여부가 깨지면 회귀 영향이 크다.
- customer-safe 차단 로직 테스트에는 최소한 다음 경우가 포함되어야 한다: 세션 없음, 프리셋 없음, helper not ready, blocked/escalated state, 정상 ready.
- 테스트는 raw helper 문자열을 직접 단언하지 말고 정규화된 DTO와 customer copy 출력만 단언해야 한다.

### 최신 기술 확인 메모

- 2026-03-20 기준 공식 Tauri 2 문서는 frontend-to-host 호출에 commands를, ordered streaming data에는 channels를 권장한다. readiness 변화는 순서가 중요한 상태 스트림이므로 global event 남용보다 command + channel 조합이 더 적합하다. 이 문장은 공식 문서에 대한 구현 해석이다. [Source: https://v2.tauri.app/develop/calling-rust/] [Source: https://v2.tauri.app/develop/calling-frontend/]
- 같은 공식 문서는 SPA 라우터에서는 event listener가 자동 해제되지 않는다고 명시한다. readiness 구독을 화면/훅에 두면 반드시 `unlisten` cleanup을 넣어 중복 구독과 이중 상태 반영을 막아야 한다. [Source: https://v2.tauri.app/develop/calling-frontend/]
- Tauri 공식 테스트 문서는 frontend test에서 `@tauri-apps/api/mocks`의 `mockIPC`를 사용할 수 있고, 각 테스트 후 `clearMocks()`로 mock 상태를 정리해야 한다고 안내한다. adapter 단위 테스트에 그대로 적용 가능하다. [Source: https://v2.tauri.app/develop/tests/mocking/]
- Zod 공식 문서는 TypeScript `strict` 모드를 요구하고, Zod 4를 최신 안정 메이저로 소개한다. readiness/shared-contracts schema는 Zod 4 기준으로 정의하는 편이 architecture의 Zod 4 요구사항과 일치한다. [Source: https://zod.dev/] [Source: https://zod.dev/v4]
