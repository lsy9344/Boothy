# Story 4.1: 드래프트 프리셋 작성과 내부 저작 작업공간

Status: done

## Story

authorized preset manager로서,
내부 저작 surface에서 드래프트 프리셋 버전을 만들고 수정하고 싶다.
그래서 booth 고객에게 저작 도구를 노출하지 않고 새 룩을 안전하게 준비할 수 있다.

## Acceptance Criteria

1. 인증된 내부 저작 세션에서 프리셋 관리자가 새 프리셋 드래프트를 만들거나 기존 드래프트 버전을 수정하면, 작업 결과는 내부 workflow 안의 draft-only preset artifact candidate로 저장되어야 한다. 또한 고객 booth catalog는 바뀌지 않아야 한다.
2. authoring surface가 열려 있을 때 드래프트 프리셋을 편집하면, UI는 권한 있는 내부 제어만 노출해야 한다. 또한 이 제어는 기본 booth customer flow에서 도달할 수 없어야 한다.

## Dev Notes

### 스토리 범위와 목적

- 이 스토리는 Epic 4 / FR-008의 첫 번째 가치 조각이다. 핵심은 승인이나 게시가 아니라, 내부 사용자만 접근 가능한 draft authoring workspace와 draft-only persistence를 만드는 것이다.
- Story 4.2의 booth compatibility validation, Story 4.3의 approval/publication, Story 4.4의 rollback/catalog version 관리는 아직 범위가 아니다. 이번 스토리에서는 draft를 만들고 다시 열어 수정할 수 있는 baseline만 고정한다.
- 고객 booth runtime은 계속 승인된 published preset만 소비해야 한다. draft는 미래 세션 준비물일 뿐, 현재 booth catalog나 active session에 영향을 주면 안 된다.

### 스토리 기반 요구사항

- FR-008은 authorized preset manager가 draft를 만들고, explicit lifecycle `draft -> validated -> approved -> published`를 거치며, booth에는 approved published preset만 노출되도록 요구한다. Story 4.1은 이 lifecycle의 `draft` baseline을 구현하는 조각이다. [Source: _bmad-output/planning-artifacts/prd.md#FR-008 Authorized Preset Authoring, Approval, and Publication]
- Epic 4의 Story 4.1 acceptance criteria는 "authenticated internal authoring session", "draft-only preset artifact candidate", "customer booth catalog unchanged", "authorized internal controls only", "default booth customer flow unreachable"를 직접 요구한다. [Source: _bmad-output/planning-artifacts/epics.md#Story 4.1: 드래프트 프리셋 작성과 내부 저작 작업공간]
- 제품의 추가 요구사항은 하나의 packaged Tauri 앱 안에 booth, operator, internal preset authoring의 capability-gated surface를 유지하고, darktable 기반 preset artifact가 권위 경로이며 고객에게 일반 편집기를 노출하지 말 것을 요구한다. [Source: _bmad-output/planning-artifacts/epics.md#Additional Requirements]

### 선행 의존성과 구현 순서

- Epic 1~3이 booth customer flow, published preset consumption, post-end guidance를 이미 고정했다. 4.1은 그 위에 authoring capability를 추가하되, booth runtime contract를 흔들지 않는 방식이 가장 안전하다.
- 권장 순서는 `draft contract -> host persistence/commands -> authoring screen/state -> capability/auth 연결 -> 테스트`다.
- Story 4.2는 4.1이 만든 draft artifact를 받아 booth compatibility validation을 수행해야 하므로, 이번 스토리에서 draft metadata와 storage shape를 느슨하게 만들면 다음 스토리에서 재작업 위험이 커진다.
- Story 4.3은 immutable published bundle을 생성해야 하므로, 4.1의 draft artifact를 published bundle과 같은 파일/스키마로 억지로 재사용하면 안 된다.

### 현재 워크스페이스 상태

- 현재 `/authoring` route는 이미 존재하고 `SurfaceAccessGuard` 뒤에 숨겨져 있지만, 실제 화면은 `PresetLibraryScreen` placeholder뿐이다. [Source: src/app/routes.tsx] [Source: src/preset-authoring/screens/PresetLibraryScreen.tsx]
- 프런트엔드 capability seam은 이미 있다. `capability-service`는 `isAdminAuthenticated`와 `allowedSurfaces`를 조합해 `authoring` 접근을 막고, `capabilities.ts`는 `booth`, `operator`, `authoring`, `settings` surface enum을 고정한다. [Source: src/app/services/capability-service.ts] [Source: src/shared-contracts/schemas/capabilities.ts]
- 그러나 런타임 capability gateway는 `get_capability_snapshot` command를 기대하는데, 현재 Rust `invoke_handler`에는 그 command가 등록되어 있지 않다. Story 4.1에서 host-owned auth/capability seam을 실제로 잇는 작업이 필요하다. [Source: src/session-domain/services/runtime-capability-gateway.ts] [Source: src-tauri/src/lib.rs]
- Rust preset 영역은 아직 published catalog 소비에만 초점이 있다. `src-tauri/src/preset/mod.rs`는 `preset_bundle`과 `preset_catalog`만 노출하며, `authoring_pipeline.rs`는 아직 없다. [Source: src-tauri/src/preset/mod.rs]
- 현재 published bundle loader는 `published-preset-bundle/v1`, `lifecycle_status == "published"`, `booth_status == "booth-safe"`인 bundle만 catalog로 인정한다. draft artifact는 이 경로를 오염시키면 안 된다. [Source: src-tauri/src/preset/preset_bundle.rs]
- 현재 worktree에는 timing/post-end/session 관련 미커밋 변경이 다수 있다. 4.1은 가능하면 `preset-authoring`, `preset`, capability/auth seam, 계약 문서 범위로 쓰기 범위를 좁히는 편이 안전하다. 이는 로컬 워크트리를 바탕으로 한 구현 추론이다.

### 기존 구현 인텔리전스

- Story 1.3과 현재 preset catalog 구현은 booth가 `presetId + publishedVersion`으로 published preset을 선택하고, catalog는 최대 6개의 booth-safe published preset만 노출하도록 이미 고정했다. 4.1은 이 published path를 건드리지 말고, 별도 draft path를 만들어야 한다. [Source: src/shared-contracts/schemas/preset-core.ts] [Source: src-tauri/src/preset/preset_catalog.rs]
- Story 2.3은 "프리셋 변경은 이후 촬영부터만 반영"을 고정했고, Story 3.3은 host-owned capability/post-end truth를 더 강화했다. 따라서 4.1에서 draft 저장이 active session preset binding이나 booth runtime state를 직접 바꾸는 설계는 회귀 위험이 크다. [Source: _bmad-output/implementation-artifacts/3-3-handoff-ready와-phone-required-보호-안내.md#구현-가드레일]
- 현재 라우팅 테스트는 기본 booth UI에서 operator/authoring/settings 링크가 보이지 않아야 하고, 인증 없는 `/authoring` 직접 진입도 `/booth`로 되돌려야 한다고 이미 잠가 두고 있다. 4.1은 이 baseline을 깨면 안 된다. [Source: src/app/routes.test.tsx]

### 구현 가드레일

- draft artifact는 published booth catalog와 물리적으로 분리해야 한다. `preset-catalog/published` 아래에 draft 파일을 저장하거나 `published-preset-bundle/v1`을 draft 저장 포맷으로 재사용하면 안 된다.
- 권장 draft root는 `appLocalData/booth-runtime/preset-authoring/drafts/{presetId}/` 계열의 별도 workspace다. 이는 현재 `sessions/`와 `preset-catalog/published/`가 base dir 아래에서 분리되는 구조를 바탕으로 한 구현 추론이다. [Source: src-tauri/src/preset/preset_catalog.rs]
- React는 authoring draft를 local state/localStorage만으로 영속화하지 말고, typed service -> Tauri command -> Rust host 검증 경로를 통해 저장해야 한다.
- authoring controls는 route 숨김만으로 보호하지 말고 capability/auth truth와 command authority까지 같이 잠가야 한다. Tauri의 runtime authority/capabilities 모델상, 보안 경계는 window/webview capability와 command allow scope까지 포함해 설계하는 편이 맞다. 마지막 문장은 공식 문서에 근거한 구현 추론이다. [Source: https://v2.tauri.app/security/runtime-authority/] [Source: https://v2.tauri.app/ko/security/capabilities/]
- Story 4.1에서는 approval, publish, rollback UI를 미리 구현하지 않는다. draft lifecycle을 억지로 `validated`나 `published`까지 진행시키면 Story 4.2/4.3 acceptance 기준이 흐려진다.
- 고객 booth surface, review flow, completion flow 어디에서도 authoring 용어, draft 상태, darktable/XMP 상세 제어를 노출하면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#FR-008 Authorized Preset Authoring, Approval, and Publication] [Source: _bmad-output/planning-artifacts/ux-design-specification.md#구속되는 UX 요구사항]

### 아키텍처 준수사항

- FR-008의 구조 매핑 중심은 `src/preset-authoring/`와 `src-tauri/src/preset/`이다. 이번 스토리는 이 범위를 중심으로 구현해야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Requirements to Structure Mapping]
- top-level route 모델은 `/booth`, `/operator`, `/authoring`, `/settings`만 허용하는 방향으로 이미 고정되어 있다. authoring 내부 단계가 늘어나더라도 top-level surface 경계는 유지하는 편이 안전하다. [Source: _bmad-output/planning-artifacts/architecture.md#Core Architectural Decisions]
- React UI는 typed adapters/services를 통해서만 native behavior에 도달해야 하고, Tauri commands만이 frontend-to-host entry point여야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Architectural Boundaries]
- `preset-authoring`은 internal preset creation/publication workflow를 소유하고, `preset-catalog`는 booth surface의 approved preset list consumption만 소유해야 한다. 두 책임을 섞지 않는다. [Source: _bmad-output/planning-artifacts/architecture.md#Architectural Boundaries]
- published preset bundle은 immutable artifact이며 booth session은 published artifact만 소비한다. 따라서 draft authoring service는 published summary schema와 별도 contract family를 가져야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Data Architecture] [Source: src/shared-contracts/schemas/preset-core.ts] [Source: src-tauri/src/preset/preset_bundle.rs]

### UX 구현 요구사항

- 고객 기본 흐름에서는 authoring 진입점이 계속 숨겨져 있어야 하며, 관리자 인증 전에는 노출되면 안 된다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#구속되는 UX 요구사항]
- Story 4.1의 내부 저작 화면은 고객용 low-density copy 규칙을 그대로 따를 필요는 없지만, booth shell처럼 보이거나 고객을 위한 편집 공간처럼 보여서는 안 된다. 내부 사용자에게 draft 상태, 저장 상태, unpublished 여부가 분명히 보여야 한다. 마지막 문장은 현재 요구와 아키텍처를 바탕으로 한 구현 추론이다.
- authoring UI는 booth catalog와 분리된 workspace라는 점을 시각적으로 분명히 해야 한다. "지금 저장해도 booth catalog는 바뀌지 않음"을 내부 사용자에게 이해 가능하게 표현하는 편이 좋다.
- darktable module-level control을 전부 재현하는 것이 Story 4.1의 목표는 아니다. 이번 스토리는 draft workspace와 draft persistence baseline이 핵심이며, richer authoring/review/publish flow는 후속 스토리로 남겨 둔다. [Source: _bmad-output/planning-artifacts/architecture.md#Darktable Capability Scope]

- 현재 로컬 기준선은 `react 19.2.4`, `react-router-dom 7.13.1`, `zod 4.3.6`, `@tauri-apps/api 2.10.1`, Rust `tauri 2.10.3`, `tauri-build 2.5.6`이다. Story 4.1은 이 기준선 안에서 구현하는 편이 안전하다. [Source: package.json] [Source: src-tauri/Cargo.toml]
- Tauri 공식 문서는 capability 파일을 `src-tauri/capabilities` 아래 JSON/TOML로 두고, window/webview별 permissions를 capability로 묶도록 안내한다. `authoring-window.json`을 booth와 별도 경계로 유지하는 것이 맞다. [Source: https://v2.tauri.app/ko/security/capabilities/]
- Tauri 공식 문서는 runtime authority가 invoke 요청 시 origin, capability, scope를 확인한 뒤 command를 전달한다고 설명한다. 따라서 authoring 보호는 숨김 route뿐 아니라 command authority까지 함께 설계해야 한다. 이것은 공식 문서를 바탕으로 한 구현 추론이다. [Source: https://v2.tauri.app/security/runtime-authority/]
- Tauri 공식 문서는 frontend가 Rust 기능에 접근할 때 `invoke` + `#[tauri::command]` + `generate_handler!` 경로를 사용하고, 여러 command는 하나의 `generate_handler!`에 함께 등록해야 한다고 설명한다. Story 4.1에서 capability snapshot이나 draft authoring command를 추가한다면 현재 `lib.rs` 등록 목록까지 함께 갱신해야 한다. [Source: https://v2.tauri.app/develop/calling-rust/]
- React 공식 문서에서 `useEffectEvent`는 effect 안에서 최신 값을 읽는 용도이며 stable identity를 제공하지 않는다. autosave, window event, host subscription에 쓰더라도 dependency 회피용으로 남용하면 안 된다. [Source: https://react.dev/reference/react/useEffectEvent]
- Zod 공식 문서는 Zod 4가 stable이라고 명시한다. draft authoring payload도 ad-hoc parsing 대신 shared schema family에 넣고 Rust에서 다시 검증하는 편이 안전하다. [Source: https://zod.dev/library-authors?id=how-to-support-zod-3-and-zod-4-simultaneously]

### 테스트 요구사항

- 최소 필수 테스트 범위는 아래와 같다.
  - 인증/권한이 없으면 `/authoring` direct navigation이 계속 차단된다.
  - 인증/권한이 있으면 draft 목록이 로드되고 새 draft 생성이 가능하다.
  - draft 저장 후 다시 열면 마지막 저장 상태가 round-trip 된다.
  - draft 저장/수정이 `preset-catalog/published`를 변경하지 않는다.
  - invalid `presetId`, 잘못된 lifecycle, 빈 이름, workspace path escape 시 host가 거부한다.
  - booth surface에는 authoring link/control/wording이 나타나지 않는다.
  - authoring command는 capability/auth 없이 호출하면 거부된다.

### 금지사항 / 안티패턴

- draft를 `preset-catalog/published` 아래에 저장하는 것 금지
- draft 저장에 `published-preset-bundle/v1` 또는 booth catalog summary schema를 그대로 재사용하는 것 금지
- route 숨김만으로 authoring 보호가 끝났다고 가정하는 것 금지
- React local state/localStorage만으로 draft persistence를 끝내고 host를 우회하는 것 금지
- Story 4.2/4.3 범위인 validation, approval, publication, rollback을 이번 스토리에서 과도하게 먼저 구현하는 것 금지
- booth customer flow, review flow, completion flow에 authoring terminology나 draft 상태를 노출하는 것 금지
