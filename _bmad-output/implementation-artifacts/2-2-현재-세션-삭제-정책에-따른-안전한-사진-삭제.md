# Story 2.2: 현재 세션 삭제 정책에 따른 안전한 사진 삭제

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a booth customer,
I want to delete only my current session photos when allowed,
so that I can clean up unwanted shots without affecting anything outside my session.

## Acceptance Criteria

1. 현재 세션 review rail에 노출된 capture가 `Current-Session Deletion Policy`에 따라 삭제 가능할 때 고객이 삭제를 확인하면, 선택된 현재 세션 capture와 그에 상관된 booth-safe source/preview/final artifact 및 manifest 참조가 제거되어야 한다. 또한 review surface에서는 해당 항목이 즉시 사라져야 하고, 다른 세션 자산에는 영향이 없어야 한다.
2. capture가 활성 host mutation 상태이거나 현재 세션 범위를 벗어나거나 post-end completion이 finalized 된 상태라서 삭제 정책상 허용되지 않을 때 고객이 삭제를 시도하면, 시스템은 삭제를 차단해야 한다. 또한 고객 화면에는 plain-language 안내만 보여야 하며 policy/storage/filesystem/internal error 용어를 노출하면 안 된다.
3. 삭제 흐름이 review rail, provider state, host command, manifest persistence 전 구간에서 재사용될 때 booth-safe asset 해석은 활성 세션 범위로만 한정되어야 한다. 따라서 delete 시도/성공/차단 전체 흐름에서 cross-session asset leakage는 0이어야 한다.

## Dev Notes

### 스토리 범위와 목적

- 이 스토리는 Epic 2 / FR-005의 두 번째 가치 조각으로, 고객이 현재 세션에서만 안전하게 사진을 정리할 수 있게 만드는 작업이다.
- Story 2.1이 current-session review rail과 session-scoped asset 해석을 잠갔다면, Story 2.2는 같은 진실 경계를 삭제까지 확장하는 단계다.
- 이 스토리는 전체 갤러리 편집기나 멀티 선택 관리 기능을 만드는 작업이 아니다. booth capture surface 안에서 bounded deletion만 추가해야 한다.

### 스토리 기반 요구사항

- `Current-Session Deletion Policy` baseline:
  - 활성 세션에 속한 capture와 상관된 artifact만 삭제 대상이 될 수 있다.
  - 삭제는 선택 capture의 source/preview/final과 manifest reference만 제거해야 한다.
  - 대상 capture가 host-owned mutation 상태이거나 post-end completion finalized 뒤면 삭제가 차단되어야 한다.
- review/delete는 current-session asset only 규칙을 공유해야 하며, cross-session leakage는 0이어야 한다.

### 선행 의존성과 구현 순서

- 선행 구현은 Story 1.5와 Story 2.1이다.
- Story 2.1에서 이미 다음 기반이 준비되어 있다.
  - 현재 세션 preview rail과 latest item 계산
  - session-scoped asset path guard
  - stale/foreign readiness 및 capture 응답 무효화
  - `SessionProvider`의 manifest 병합 및 customer-safe readiness 정규화
- 따라서 2.2 구현은 새로운 전역 이미지 캐시나 별도 gallery truth를 만들지 말고, manifest/capture/service/provider 경계를 그대로 확장해야 한다.

### 현재 워크스페이스 상태

- 현재 프런트엔드는 `CaptureScreen` + `LatestPhotoRail` + `SessionProvider` 조합으로 review rail을 그린다.
- typed host 호출은 `start_session`, `select_active_preset`, `get_capture_readiness`, `request_capture`까지만 구현되어 있고, deletion command는 아직 없다.
- Rust host는 `SessionPaths`, `session_repository`, `capture/normalized_state`, `capture_commands`를 통해 session manifest와 capture persistence를 다룬다.

### 이전 스토리 인텔리전스

- Story 2.1은 `selectCurrentSessionPreviews`, `isSessionScopedAssetPath`, `SessionProvider` sanitization을 통해 현재 세션 외 자산이 review rail에 들어오지 못하게 잠갔다.
- Story 2.1 후속 리뷰에서 foreign readiness, wrong-root asset path, stale success payload, preview scrub fallback 같은 경계 조건이 여러 번 수정되었다. Story 2.2도 동일한 종류의 foreign/stale delete 응답을 반드시 경계해야 한다.
- 현재 rail은 `finalReady` capture라도 safe preview가 남아 있으면 계속 노출한다. 삭제 후에는 이 latest continuity가 깨지지 않도록 다음 item 재계산이 필요하다.
- 최근 기준선 커밋은 `d4a5c15 feat: implement story 1-5 truthful preview waiting flow`, `09741aa chore: mark story 1-5 done`이며, Story 2.1 작업은 아직 워킹트리 기반으로 이어지고 있다.

### 구현 가드레일

- 활성 세션 durable truth는 session root와 `session.json`이다. deletion도 이 truth를 변경해야 하며 프런트 전용 상태만 지우는 방식으로 끝내면 안 된다. [Source: _bmad-output/planning-artifacts/epics.md#Additional Requirements]
- React는 raw host 호출을 JSX에서 직접 만들지 말고 기존 typed adapter/service/provider를 통해 상태를 소비해야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Frontend Architecture]
- 삭제는 review rail에 보이는 current-session asset 범위 안에서만 동작해야 하며, 다른 세션 또는 authoring/runtime 외부 상태를 건드리면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#FR-005 Current-Session Review, Deletion, and In-Session Preset Change]
- 고객 문구에는 policy/storage/filesystem/render-engine/diagnostic 같은 내부 용어를 노출하면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#NFR-001 Customer Guidance Density and Simplicity]

### 아키텍처 준수사항

- review/delete capability는 `src/booth-shell/`, `src/session-domain/`, `src/capture-adapter/`, `src-tauri/src/capture/`, `src-tauri/src/session/` 경계 안에서 풀어야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Requirements to Structure Mapping]
- frontend-to-host request path는 Tauri commands가 기준선이며, session-scoped persistence/deletion/privacy isolation은 host에서 집행해야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Data Architecture]
- shared contract는 TypeScript `src/shared-contracts/`와 Rust `src-tauri/src/contracts/`가 같은 family로 유지해야 한다. deletion contract도 이 패턴을 따라야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Project Structure & Boundaries]

### UX 구현 요구사항

- 삭제 액션은 현재 세션 rail 안에서만 노출되어야 하고, 별도 편집 화면처럼 보이면 안 된다. [Source: _bmad-output/planning-artifacts/epics.md#UX-DR13]
- 차단 상태에서는 고객이 이해할 수 있는 plain-language 안내를 사용해야 하고, 내부 정책명이나 스토리지 세부사항은 숨겨야 한다. [Source: _bmad-output/planning-artifacts/epics.md#Story 2.2: 현재 세션 삭제 정책에 따른 안전한 사진 삭제]
- 삭제 성공 후에는 해당 item이 rail에서 즉시 사라지고, 최신 사진 표시가 남은 current-session item 기준으로 자연스럽게 갱신되어야 한다.

### 최신 기술 확인 메모

- 현재 로컬 패키지 기준선은 `react 19.2.4`, `react-router-dom 7.13.1`, `zod 4.3.6`, `@tauri-apps/api 2.10.1`이다. 이 스토리는 같은 메이저/마이너 기준선 안에서 구현하고 새 의존성을 추가하지 않는 편이 안전하다.
- Tauri 공식 문서는 현재도 SPA 프런트엔드로 Vite 구성을 권장하고, frontend-host 통신은 Tauri command/invoke 경계를 기준으로 둔다. deletion도 같은 request/response 패턴 안에서 구현하는 것이 안전하다. [Source: https://v2.tauri.app/start/frontend/vite/]
- Zod 공식 문서는 Zod 4가 stable임을 명시한다. deletion contract도 기존 shared schema family 안에 추가해야 하며 별도 ad-hoc validation을 만들지 않는 편이 좋다. [Source: https://zod.dev/v4/versioning?id=why-cant-libraries-just-support-v3-and-v4-simultaneously]

### 테스트 요구사항

- 최소 필수 테스트 범위:
  - deletable current-session capture 삭제 성공 시 manifest와 rail에서 제거된다.
  - correlated preview/final/original만 삭제되고 다른 current-session capture는 유지된다.
  - foreign session capture 또는 foreign asset path는 삭제 대상이 되지 않는다.
  - `previewWaiting` 같은 active mutation target은 차단되고 파일/manifest가 변하지 않는다.
  - stale 또는 foreign delete 응답은 현재 세션 state를 오염시키지 않는다.
  - 고객 안내 문구에 internal terminology가 나타나지 않는다.

### 금지사항 / 안티패턴

- 전역 이미지 목록이나 raw filesystem scan으로 삭제 대상을 찾는 것 금지
- 현재 세션 rail에 없는 foreign capture를 UI에서 직접 식별해 삭제하는 것 금지
- 삭제 성공을 프런트 state만 갱신하고 durable manifest/file 삭제 없이 끝내는 것 금지
- 삭제 차단 사유를 내부 policy code, path, host error 원문으로 그대로 노출하는 것 금지
- Story 2.3 범위인 future preset change 또는 Story 3.x post-end UX를 미리 확장하는 것 금지
