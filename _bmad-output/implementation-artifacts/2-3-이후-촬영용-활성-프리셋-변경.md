# Story 2.3: 세션 중 활성 프리셋 자유 변경

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

booth customer로서,
내 세션 중에도 활성 프리셋을 바꾸고 싶다.
그래서 이미 찍은 사진은 그대로 두고, 다음 촬영부터 원하는 승인된 룩으로 이어갈 수 있다.

## Acceptance Criteria

1. 활성 세션에서 이미 선택된 프리셋이 있을 때 고객이 세션 중 어느 시점에서든 다른 승인된 게시 프리셋을 고르면, 새 프리셋은 세션의 활성 프리셋으로 즉시 반영되어야 한다. 또한 이미 저장된 현재 세션 자산은 촬영 시점에 사용된 프리셋 바인딩을 유지해야 한다.
2. 고객이 capture, review, 또는 preset-selection 관련 surface에서 프리셋을 바꾸면, UI는 다음 촬영에 적용될 새 활성 프리셋을 분명하게 보여줘야 한다. 또한 이전 촬영본이 다시 보정되었거나 다시 묶였다는 인상을 주면 안 된다.
3. 고객이 프리셋 전환을 요청했지만 해당 프리셋이 더 이상 사용 가능하지 않거나 세션에 안전하게 바인딩할 수 없으면, 이전 활성 프리셋은 그대로 유지되어야 한다. 또한 고객에게는 현재 프리셋을 유지하거나 다른 승인된 프리셋을 고르라는 plain-language 안내만 보여줘야 한다.
4. 프리셋 전환이 성공하면, 활성 프리셋 변경 확인은 승인된 하드웨어에서 1초 이내에 인지되어야 한다. 또한 전환으로 인해 이미 저장된 현재 세션 자산이 변형되면 안 된다.

## Dev Notes

### 스토리 범위와 목적

- 이 스토리는 Epic 2 / FR-005의 세 번째 가치 조각이다. 핵심은 고객이 세션 중에도 룩을 바꾸되, 이미 찍은 사진은 그대로 두고 다음 촬영부터만 새 룩이 적용된다는 신뢰를 만드는 것이다.
- 이 스토리는 새 편집기나 세션 재처리 기능을 추가하는 작업이 아니다. 부스 안에서 승인된 preset catalog를 다시 열고, 현재 세션의 active preset binding만 안전하게 바꾸는 작업이다.
- Story 2.1과 2.2가 current-session review/delete 경계를 잠갔다면, Story 2.3은 같은 경계 안에서 "forward-only preset change"를 추가하는 단계다.

### 스토리 기반 요구사항

- 고객은 세션 중 언제든 active preset을 바꿀 수 있어야 한다. [Source: _bmad-output/planning-artifacts/prd.md#FR-005 Current-Session Review, Deletion, and In-Session Preset Change]
- 각 프리셋 변경은 이후 캡처에만 적용되고, 이미 저장된 자산은 다시 쓰면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#FR-005 Current-Session Review, Deletion, and In-Session Preset Change]
- booth runtime은 승인된 게시 프리셋만 다뤄야 하고, 고객에게는 direct editing workspace나 darktable 용어를 보여주면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#Customer Preset Experience Baseline]
- active session은 preset version을 명시적으로 참조해야 하며, preset-authoring이나 publication은 미래 세션에만 영향을 줘야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Core Architectural Decisions]

### 선행 의존성과 구현 순서

- 선행 구현은 Story 1.3, Story 1.5, Story 2.1, Story 2.2다.
- Story 1.3에서 이미 preset catalog loading과 `select_active_preset` 저장 경로가 존재한다.
- Story 2.1과 2.2에서 review rail, current-session selector, stale/foreign response guard, deletion policy 경계가 정리되었다.
- 따라서 2.3 구현은 새 preset subsystem을 만드는 것이 아니라 기존 catalog/service/provider/host 경계를 세션 중 변경 시나리오까지 확장하는 방식이어야 한다.

### 현재 워크스페이스 상태

- `PresetSelectScreen`은 현재 session start 이후의 초기 preset 선택용으로 구현되어 있고, 성공 시 `flowStep: 'capture'`로 이동한다.
- `CaptureScreen`과 `ReadinessScreen`은 현재 active preset 이름과 촬영 액션만 보여 주며, in-session preset change entry는 아직 없다.
- `SessionProvider`의 `selectActivePreset` 실패 처리에는 `preset-not-available` 시 `resetToPresetSelection()`이 포함되어 있어, 세션 중 변경 시나리오에는 그대로 쓰면 현재 룩이 지워지는 회귀 위험이 있다.
- Rust host의 `select_active_preset`은 `session.json`의 `activePreset`만 갱신하고 기존 `captures`는 건드리지 않으므로, 이 경계는 재사용 가치가 높다.

### 이전 스토리 인텔리전스

- Story 2.1은 현재 세션 review rail과 session-scoped asset guard를 안정화했다. preset change가 들어와도 다른 세션 자산 노출 금지 규칙은 그대로 유지해야 한다. [Source: _bmad-output/implementation-artifacts/2-1-현재-세션-사진-레일과-세션-범위-검토.md#구현-가드레일]
- Story 2.2는 deletion path에서도 stale/foreign 응답이 현재 세션을 덮지 못하게 만들었다. preset change도 같은 종류의 late response/foreign response 방어가 필요하다. [Source: _bmad-output/implementation-artifacts/2-2-현재-세션-삭제-정책에-따른-안전한-사진-삭제.md#이전-스토리-인텔리전스]
- 최근 커밋은 Story 1.5 baseline 중심(`d4a5c15`, `09741aa`)이라, Story 2.x의 실질 기준선은 git보다 현재 workspace 코드와 implementation artifact 문서 쪽에 더 많이 남아 있다.

### 구현 가드레일

- `resetToPresetSelection()`은 현재 `selectedPreset`를 비우고 preset catalog를 초기화한다. 세션 중 프리셋 변경 실패에서는 이 경로를 그대로 쓰지 말고, "현재 룩 유지 + 재선택 안내" 경로를 별도로 두어야 한다.
- preset change 성공은 `manifest.activePreset`만 앞으로의 기준선으로 바꾸는 작업이어야 한다. 이미 저장된 capture asset, capture ordering, deletion eligibility, preview readiness는 바꾸면 안 된다.
- 현재 `SessionCaptureRecord`는 `activePresetVersion`만 가진다. Story 2.3에서 고객에게 과거 캡처의 룩을 구분해 보여 주거나 "촬영 당시 바인딩 유지"를 계약 수준으로 잠그려면, capture-level preset identity가 충분한지 먼저 확인해야 한다.
- current-session rail과 customer copy는 "현재 활성 룩"과 "기존 사진이 찍힌 룩"을 분리해서 표현해야 한다. 이전 사진이 새 룩으로 재적용된 것처럼 읽히는 문구는 금지다.
- booth UI는 여전히 current-session만 다뤄야 하며, preset change 중에도 cross-session asset leakage는 0이어야 한다. [Source: _bmad-output/planning-artifacts/prd.md#NFR-004 Session Isolation and Privacy]

### 아키텍처 준수사항

- FR-005 구현 경계는 `src/booth-shell/`, `src/session-domain/`, `src-tauri/src/session/`를 중심으로 유지한다. [Source: _bmad-output/planning-artifacts/architecture.md#Requirements to Structure Mapping]
- React UI는 typed adapter/service를 통해서만 호스트 기능에 접근해야 하며, JSX에서 직접 `invoke`를 호출하면 안 된다. [Source: _bmad-output/planning-artifacts/architecture.md#Pattern Examples]
- active sessions는 preset versions를 명시적으로 참조해야 하고, preset-authoring은 active booth session data를 직접 수정하지 않는다. [Source: _bmad-output/planning-artifacts/architecture.md#Data Architecture]
- `session.json`과 shared contract family는 양쪽 언어에서 함께 유지돼야 한다. capture binding 정보를 확장할 경우 `src/shared-contracts/`, `src-tauri/src/contracts/`, `src-tauri/src/session/`를 동시에 맞춰야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Closed Contract Freeze Baseline]

### UX 구현 요구사항

- 고객은 세션 중에도 룩을 바꿀 수 있어야 하지만, direct editing처럼 느껴지면 안 된다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Reading Guide]
- 고객은 현재 활성 프리셋과 현재 세션 범위의 최신 사진을 항상 이해할 수 있어야 한다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Reading Guide]
- 주요 고객 문구는 plain-language와 낮은 문구 밀도 원칙을 지켜야 하며, darktable/XMP/style/authoring 같은 내부 용어를 노출하면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#NFR-001 Customer Guidance Density and Simplicity]
- preset switch 확인은 "다음 촬영부터 적용"이라는 의미를 주고, 이전 촬영본 재처리처럼 읽히는 성공 카피를 쓰면 안 된다.

### 최신 기술 확인 메모

- 현재 로컬 기준선은 `react 19.2.4`, `react-router-dom 7.13.1`, `@tauri-apps/api 2.10.1`, `zod 4.3.6`이다. Story 2.3은 새 의존성 없이 이 기준선 안에서 구현하는 편이 안전하다. [Source: package.json]
- React Router의 current data-router 가이드는 `createBrowserRouter`와 `RouterProvider` 패턴을 계속 사용한다. preset switch는 새 라우터 체계를 들여오는 문제가 아니라 기존 `/booth` state 흐름 안에서 해결해야 한다. [Source: https://reactrouter.com/start/data/installation]
- Tauri의 Vite 프런트엔드 가이드는 기존 SPA/Vite + command boundary 구성을 전제로 한다. preset switch도 동일한 command/service 경계를 유지해야 한다. [Inference from official guide: https://v2.tauri.app/start/frontend/vite/]
- Zod 공식 문서는 TypeScript-first schema validation과 strict mode를 강조한다. capture binding contract를 바꾼다면 ad-hoc shape를 만들지 말고 shared schema family를 먼저 고쳐야 한다. [Source: https://zod.dev/]

### 테스트 요구사항

- 최소 필수 테스트 범위:
  - capture screen에서 현재 룩 변경 entry가 보이고, 성공 후 새 active preset이 즉시 표시된다.
  - preset switch 실패(`preset-not-available`, stale, foreign session) 시 이전 active preset이 유지된다.
  - preset switch 후 이후 캡처만 새 바인딩을 사용하고, 기존 capture record와 rail 표시는 유지된다.
  - review rail과 readiness summary가 현재 활성 룩과 과거 캡처 바인딩을 혼동하지 않는다.
  - 고객 화면에 internal terminology가 노출되지 않는다.

### 금지사항 / 안티패턴

- in-session preset switch 실패 시 `selectedPreset`를 `null`로 지우는 것 금지
- preset change를 위해 기존 current-session review rail이나 manifest captures를 초기화하는 것 금지
- 과거 캡처를 새 active preset 기준으로 재분류하거나 rail 문구를 덮어써 재편집처럼 보이게 만드는 것 금지
- JSX나 component에서 직접 새 `invoke('select_active_preset')` 호출을 추가하는 것 금지
- 승인되지 않은 preset, hidden preset, authoring draft를 고객 선택지에 노출하는 것 금지
