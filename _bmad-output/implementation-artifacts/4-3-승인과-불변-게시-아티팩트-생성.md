## Story

authorized preset manager로서,
검증이 끝난 프리셋을 승인하고 불변의 버전드 게시 아티팩트로 만들고 싶다.
그래서 미래 부스 세션이 안정적이고 추적 가능한 프리셋 카탈로그 엔트리를 사용할 수 있다.

## Acceptance Criteria

1. 프리셋 버전이 `validated` 상태일 때 authorized approver가 승인 후 게시하면, 시스템은 stable identity, version, catalog metadata를 가진 immutable published preset artifact bundle을 만들어야 한다. 또한 프리셋 lifecycle은 `approved`를 거쳐 `published`로 전진해야 한다.
2. 프리셋이 게시된 뒤 future session이 booth catalog를 로드하면, 새 published preset이 selectable catalog item으로 나타날 수 있어야 한다. 또한 publication event가 active session을 mutate하면 안 된다.
3. publication 요청 시 validation이 stale하거나, 필수 artifact metadata가 호환되지 않거나, publication이 immutability 또는 future-session-only 규칙을 위반하면 publication은 거절되어야 한다. 또한 `published` artifact는 생성되면 안 되고, 프리셋은 이전 lifecycle state를 유지해야 하며, authorized user는 actionable rejection guidance를 받아야 한다.
4. publication 요청이 거절되면, 시스템은 rejected action, reason, actor, timestamp를 audit history에 기록해야 한다. 또한 booth catalog와 active session은 그대로 유지되어야 한다.

## Dev Notes

### 스토리 범위와 목적

- 이 스토리는 Epic 4 / FR-008의 세 번째 가치 조각이다. 핵심은 "게시 버튼 추가"가 아니라, booth runtime이 실제로 신뢰할 수 있는 immutable published preset artifact를 승인 경계를 거쳐 만들고 future-session-only로 노출하는 것이다.
- Story 4.1이 draft authoring workspace, Story 4.2가 booth compatibility validation이라면, Story 4.3은 validated draft를 booth-consumable published bundle로 승격시키는 첫 계약이다.
- 이 스토리는 active session의 현재 프리셋 바인딩을 다시 쓰는 작업이 아니다. publication은 미래 세션 catalog만 바꿔야 한다.

### 스토리 기반 요구사항

- authorized preset managers는 explicit lifecycle `draft -> validated -> approved -> published`를 따라야 한다. [Source: _bmad-output/planning-artifacts/prd.md#FR-008 Authorized Preset Authoring, Approval, and Publication]
- publication은 immutable published artifact bundle과 catalog-facing metadata를 만들고, booth session은 approved published artifact만 소비해야 한다. [Source: _bmad-output/planning-artifacts/prd.md#Published Preset Artifact Model]
- publication과 rollback은 future session에만 영향을 줘야 하며 active session data 또는 active session preset binding을 mutate하면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#MVP Scope Clarifications]
- publication rejection은 no-op이어야 하고 rejected action, reason, actor, timestamp가 audit history에 남아야 한다. [Source: _bmad-output/planning-artifacts/epics.md#Story 4.3: 승인과 불변 게시 아티팩트 생성]

### 구현 가드레일

- 새 publish flow가 booth runtime이 이미 읽는 `published-preset-bundle/v1`과 다른 두 번째 published format을 만들면 안 된다.
- immutable publication은 기존 published version directory를 수정하는 방식이 아니라, 새 version directory를 원자적으로 만드는 방식이어야 한다.
- publication success가 global catalog root를 바꾸더라도 active session manifest, current-session catalog snapshot, current capture binding은 즉시 바뀌면 안 된다.
- validation이 stale하거나 artifact metadata가 incomplete하면 "거의 게시" 상태를 남기지 말고 이전 lifecycle state로 되돌린 채 명시적으로 거절해야 한다.
- draft, validated, approved internal metadata나 darktable raw editor jargon이 booth customer surface로 새어 나오면 안 된다.
- audit history는 bundle truth와 분리돼야 한다. bundle 디렉터리만으로 rejected attempt를 추적하려 들면 안 된다.
- publication이 실패해도 booth catalog loader가 malformed half-written bundle 때문에 전체 catalog를 망가뜨리면 안 된다.

### 최신 기술 확인 메모

- 현재 로컬 기준선은 `react 19.2.4`, `react-router-dom 7.13.1`, `@tauri-apps/api 2.10.1`, `zod 4.3.6`이다. Story 4.3은 새 프레임워크보다 기존 typed boundary 위에서 authoring publication을 확장하는 편이 안전하다. [Source: package.json]
- Tauri 공식 문서는 command를 separate module에 둘 때 `pub`로 선언해야 하고 command name은 module scope를 무시하므로 전역에서 unique해야 한다고 설명한다. 새 publication command 추가 시 그대로 따라야 한다. [Source: https://v2.tauri.app/develop/calling-rust/]
- Tauri 공식 문서는 ordered stream에는 channels가 event system보다 적합하다고 설명한다. Story 4.3이 단발 publish 응답만 다루면 command response로 충분하지만, 나중에 publish progress stream을 넣는다면 event 남발보다 channel을 먼저 검토하는 편이 안전하다. 이것은 공식 문서와 현재 구조를 바탕으로 한 구현 추론이다. [Source: https://v2.tauri.app/develop/calling-frontend/]
- Tauri 공식 문서는 `mockIPC(..., { shouldMockEvents: true })`로 Rust emit 이벤트를 테스트에서 시뮬레이션할 수 있다고 안내한다. publish 완료나 rejection 이벤트를 프런트 테스트에 연결할 때 활용 가능하다. [Source: https://v2.tauri.app/develop/tests/mocking/]
- Zod 공식 사이트는 Zod를 TypeScript-first schema validation 라이브러리로 안내하고, 현재 Zod 4 baseline 위에서 immutable schema API를 유지한다. publication payload와 result도 ad-hoc parsing 대신 shared schema family로 추가하는 편이 안전하다. [Source: https://zod.dev/]

### 테스트 요구사항

- 최소 필수 테스트 범위:
  - validated가 아닌 draft는 publish할 수 없다.
  - stale validation timestamp 또는 incompatible metadata는 published artifact를 만들지 못한다.
  - publish 성공 시 `preset-catalog/published/{presetId}/{publishedVersion}/bundle.json`이 booth loader가 읽을 수 있는 shape로 생성된다.
  - duplicate version publish는 기존 directory를 mutate하지 않고 거절된다.
  - rejection은 audit history에 남고, booth catalog와 active session manifest는 그대로 유지된다.
  - active session이 이미 가진 preset binding과 current-session capture는 publication event 뒤에도 변하지 않는다.
  - future session catalog load에서는 새 published preset이 selectable item으로 나타난다.
  - authoring capability가 없으면 publish UI와 command entry가 노출되지 않는다.

### 금지사항 / 안티패턴

- published bundle shape를 booth loader와 다른 포맷으로 따로 만드는 것 금지
- existing published version directory를 in-place edit해서 immutability를 깨는 것 금지
- publication 성공 시 active session manifest나 existing capture record를 업데이트하는 것 금지
- rejection 후 partial directory, broken `bundle.json`, dangling preview path를 남기는 것 금지
- booth customer surface에서 authoring draft/validated/approved 상태를 직접 노출하는 것 금지
- authoring UI에서 직접 `invoke`를 호출하고 shared-contracts 검증을 건너뛰는 것 금지
- rejected publication을 bundle 디렉터리 naming만으로 추적하려는 것 금지
