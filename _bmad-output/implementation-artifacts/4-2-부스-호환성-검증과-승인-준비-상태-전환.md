## Story

authorized preset manager로서,
승인 전에 드래프트 프리셋이 부스에서 안전하고 재현 가능하게 동작하는지 검증하고 싶다.
그래서 호환성 기준을 통과한 프리셋만 다음 승인 단계로 올릴 수 있다.

## Acceptance Criteria

1. 드래프트 프리셋 버전이 있을 때 관리자가 부스 호환성 검증을 실행하면, 시스템은 required render compatibility와 artifact rule을 평가해야 한다. 그리고 모든 필수 검증이 통과한 경우에만 드래프트가 `draft`에서 `validated`로 이동할 수 있어야 한다.
2. 드래프트 프리셋이 검증에 실패했을 때 결과가 반환되면, 해당 프리셋은 고객 카탈로그 밖에 남아 있어야 한다. 또한 내부 사용자는 active session에 영향을 주지 않는 범위에서 actionable validation feedback을 볼 수 있어야 한다.

## Dev Notes

### 스토리 범위와 목적

- 이 스토리는 Epic 4 / FR-008의 두 번째 가치 조각이다. 핵심은 "프리셋을 게시한다"가 아니라, 게시 전에 booth-safe compatibility를 host-owned truth로 검증하고 `validated` 상태를 만드는 것이다.
- Story 4.3에서 다룰 approval/publication보다 한 단계 앞선 경계만 만든다. 따라서 이번 스토리에서는 immutable published artifact 생성이나 booth catalog mutation을 완료로 간주하면 안 된다.
- 검증은 authoring 품질 검토일 뿐 customer booth state를 바꾸는 행위가 아니다. 고객이 보는 프리셋 카탈로그는 계속 approved published preset만 유지되어야 한다.

### 스토리 기반 요구사항

- authorized preset manager는 draft preset을 booth compatibility 기준으로 검증한 뒤에만 `validated` 상태로 올릴 수 있어야 한다. [Source: _bmad-output/planning-artifacts/epics.md#Story 4.2: 부스 호환성 검증과 승인 준비 상태 전환]
- failed validation은 active sessions를 건드리지 않은 채 actionable feedback을 보여줘야 한다. [Source: _bmad-output/planning-artifacts/epics.md#Story 4.2: 부스 호환성 검증과 승인 준비 상태 전환]
- FR-008은 lifecycle state를 `draft`, `validated`, `approved`, `published`로 분리하고, booth runtime은 approved published preset만 소비하도록 고정한다. [Source: _bmad-output/planning-artifacts/prd.md#FR-008 Authorized Preset Authoring, Approval, and Publication]
- published artifact는 stable identity, render compatibility, booth-safe preview/final behavior를 포함해야 한다. Story 4.2의 validation은 이 published contract를 만족할 준비가 되었는지를 미리 확인해야 한다. [Source: _bmad-output/planning-artifacts/prd.md#Published Preset Artifact Model]
- publication과 rollback은 future sessions only다. 따라서 validation 역시 active session preset binding이나 current-session assets를 절대 변경하면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#Authoring, Approval, and Publication]

### 선행 의존성과 구현 순서

- 권장 구현 순서는 Story 4.1 -> Story 4.2 -> Story 4.3 -> Story 4.4다.
- 다만 현재 implementation artifacts에는 Story 4.1 문서가 아직 없고, 코드베이스에도 authoring은 placeholder 수준이다. 따라서 4.2를 먼저 구현하더라도 최소 draft storage/lifecycle 토대는 이번 스토리 안에서 같이 정리하거나 4.1 작업 브랜치와 함께 통합해야 한다.
- 이번 스토리의 완료 기준은 `draft -> validated` 전이와 actionable feedback까지다. approval/publish 버튼, immutable bundle finalization, rollback UX는 Story 4.3 / 4.4로 넘긴다.

### 현재 워크스페이스 상태

- `/authoring` route와 capability gating은 이미 존재하지만, `PresetLibraryScreen`은 현재 placeholder다. [Source: src/app/routes.tsx] [Source: src/preset-authoring/screens/PresetLibraryScreen.tsx]
- shared contracts는 아직 booth runtime이 쓰는 published preset summary와 preset selection까지만 다루고 있다. draft artifact, validation request/result, lifecycle transition schema는 없다. [Source: src/shared-contracts/schemas/preset-core.ts] [Source: src/shared-contracts/dto/preset.ts]
- Rust host에는 published preset catalog loader와 bundle parser만 있다. 현재는 `published-preset-bundle/v1`, `published`, `booth-safe` artifact만 읽도록 설계되어 있어, draft/validated 경계는 별도 추가가 필요하다. [Source: src-tauri/src/preset/preset_bundle.rs] [Source: src-tauri/src/preset/preset_catalog.rs]
- docs/contracts에는 `session.json`만 있고 preset bundle contract와 authoring publication contract 문서는 아직 없다. 아키텍처는 이것들을 closed contract baseline으로 명시하지만 실제 문서화는 미완이다. [Source: docs/contracts/session-manifest.md] [Source: _bmad-output/planning-artifacts/architecture.md#Closed Contract Freeze Baseline]
- existing booth flow는 이미 published preset을 세션에 바인딩하고, top-6 booth-safe catalog만 노출하는 보호 로직을 갖고 있다. Story 4.2는 이 보호 경계를 깨뜨리면 안 된다. [Source: src-tauri/tests/session_manifest.rs]

### 구현 가드레일

- React authoring UI가 자체적으로 "통과/실패"를 계산하지 말고, host가 만든 validation result를 그대로 표시해야 한다.
- `validated`는 내부 approval-ready 상태지, customer-visible published 상태가 아니다. `validated` artifact를 booth catalog에 흘려 넣으면 안 된다.
- validation finding은 수정 가능한 형태로 요약해야 하지만, 고객 화면에서 금지된 darktable/module/internal path 용어를 고객 surface로 새어 나가게 하면 안 된다.
- dry-run render를 도입하더라도 active booth render queue와 충돌하지 않는 authoring 전용 configdir/library/working directory를 사용해야 한다.
- preset validation이 session manifest, capture record, activePreset binding, current booth alias 같은 current-session truth를 만지면 안 된다.

### 아키텍처 준수사항

- preset-authoring은 active booth session data를 직접 편집하지 않고 future preset versions만 생산해야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Data Architecture]

### UX 구현 요구사항

- authoring 진입점은 고객 기본 흐름에서 숨겨져 있어야 하고, 관리자 비밀번호 인증 전에는 시각적으로도 노출되면 안 된다. [Source: _bmad-output/planning-artifacts/epics.md#UX-DR9]
- 고객용 문구 저밀도 원칙은 booth/customer surface 기준이지만, authoring surface 역시 booth-safe customer copy와 internal validation copy를 섞지 않도록 분리해야 한다. [Source: _bmad-output/planning-artifacts/epics.md#UX-DR5]
- authoring UI는 내부용이라도 accessibility baseline을 따라야 하며, validation detail modal이나 drawer를 쓰면 focus management와 ESC 닫기를 지원해야 한다. [Source: _bmad-output/planning-artifacts/epics.md#UX-DR16]
- draft validation 실패 화면은 "무엇이 빠졌는지", "어떤 경계를 못 통과했는지", "다음에 뭘 고쳐야 하는지"를 빠르게 스캔할 수 있어야 한다. raw JSON 전체를 뿌리는 형태는 피한다.

### 최신 기술 확인 메모

- 현재 로컬 baseline은 `react 19.2.4`, `react-router-dom 7.13.1`, `@tauri-apps/api 2.10.1`, `zod 4.3.6`이다. Story 4.2는 새 프레임워크를 들이기보다 existing stack 안에서 authoring contract와 validation flow를 확장하는 편이 안전하다. [Source: package.json]
- Zod 공식 문서는 2025-07-08 기준 package root가 Zod 4를 export하며, Zod 4가 stable이라고 명시한다. 현재 repo의 `zod ^4.3.6` baseline은 이 방향과 맞으므로 새 schema는 shared contract 계층에 계속 모으는 것이 타당하다. [Source: https://zod.dev/v4/versioning?id=update--july-8th-2025]
- Tauri v2 공식 보안 문서는 capability 기반 surface 제한을 지속적으로 강조한다. authoring validation 기능을 추가하더라도 booth window 권한과 authoring window 권한을 섞지 않는 편이 안전하다. 이것은 공식 capability 문서를 바탕으로 한 구현 추론이다. [Source: https://v2.tauri.app/security/capabilities/]
- darktable reference는 현재 pinned release를 `5.4.1`로 고정하고 있으며, version pin을 올릴 때 preset revalidation이 필요하다고 명시한다. Story 4.2의 compatibility validation은 이 version pin drift를 조기에 막는 역할을 해야 한다. [Source: reference/darktable/README.md]

### 테스트 요구사항

- 최소 필수 검증 범위:
  - required draft metadata 누락 시 validation이 실패하는지
  - `presetId`, version, preview asset path, darktableVersion 형식 불일치 시 actionable finding이 반환되는지
  - preview asset path가 draft root 밖으로 escape하면 거부되는지
  - dry-run 또는 동등 render compatibility check 실패 시 lifecycle이 `validated`로 올라가지 않는지
  - validation success 후에도 booth published catalog 결과가 바뀌지 않는지
  - authoring surface가 capability/auth 없이 렌더되지 않는지
  - 실패 결과가 active session, booth alias, current capture state를 오염시키지 않는지

### 금지사항 / 안티패턴

- `validated` preset을 `published`와 같은 의미로 취급하는 것 금지
- booth UI가 draft/validated artifact를 읽도록 catalog loader를 느슨하게 바꾸는 것 금지
- validation 결과를 React only rule로 판정하고 Rust host 재검증을 생략하는 것 금지
- active booth render queue나 active session manifest를 authoring validation이 직접 건드리는 것 금지
- darktable GUI 상태나 `.dtstyle` 이름만으로 booth compatibility를 통과시켰다고 판단하는 것 금지
- validation 실패 메시지에 customer-facing copy budget을 깨는 장황한 raw diagnostics를 그대로 노출하는 것 금지
