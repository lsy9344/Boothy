# Story 2.5: 5분 경고와 종료 시각 알림, 촬영 가능 여부 갱신

Status: done

## Story

booth customer로서,
세션 종료 5분 전 경고와 정확 종료 시점이 믿을 수 있게 동작하고 지금 촬영 가능한지 즉시 이해하고 싶다.
그래서 마지막 촬영을 서두르더라도 혼란 없이 마무리하고 종료 직후 다음 안내를 자연스럽게 받아들일 수 있다.

## Acceptance Criteria

1. 활성 세션이 종료 5분 전 임계값에 도달하면 host가 현재 세션 기준으로 경고 이벤트를 확정하고, 승인된 사운드 경고와 시각 배지가 정확히 한 번 함께 보여야 한다. 또한 고객은 이 시점에도 촬영 가능 여부를 plain-language로 이해할 수 있어야 한다.
2. 운영 환경의 qualifying session을 기준으로 warning/end 임계값을 평가할 때, 5분 경고와 정확 종료 알림은 99% 세션에서 +/- 5초 이내에 발생해야 한다. 또한 허용 오차를 벗어난 세션은 후속 분석을 위해 감사 가능하게 기록되어야 한다.
3. 세션이 조정된 종료 시각에 도달하면 host는 stale client state와 무관하게 새 촬영을 즉시 차단해야 한다. 또한 승인된 종료 사운드와 시각 안내가 함께 제공되어야 하고, 승인된 연장 적용과 로그 기록이 없는 한 촬영 가능 상태로 되돌아가면 안 된다.
4. exact-end 직후 explicit post-end truth가 아직 확정되지 않은 짧은 구간이 있다면 고객 화면은 "촬영이 끝났고 다음 단계를 준비 중"이라는 중립 안전 문구만 보여야 한다. 이미 `Export Waiting`, `Completed`, `Phone Required` 중 하나의 explicit post-end truth가 확정된 뒤에는 generic ended/waiting copy가 그 상태를 되돌리거나 희석하면 안 된다.

## Dev Notes

### 스토리 범위와 목적

- Epic 2의 원래 의도는 고객이 경고 시점과 종료 시점을 믿고 행동하도록 만드는 것이다.
- 현재 workspace에는 Story 2.4와 Story 3.1의 구현 흔적이 이미 존재하므로, 이번 스토리의 실제 역할은 "warning/exact-end baseline을 다시 만들기"가 아니라 timing truth와 post-end truth 사이의 제품 경계를 흔들리지 않게 고정하는 것이다.
- 핵심은 세 가지다.
  - 5분 경고가 세션당 정확히 한 번만 고객에게 전달될 것
  - exact-end 이후 stale client state가 있어도 새 촬영이 열리지 않을 것
  - explicit post-end state가 생긴 뒤 generic ended copy가 다시 앞에 나오지 않을 것

### 스토리 기반 요구사항

- FR-006은 조정된 종료 시각, 5분 경고, exact-end behavior를 booth customer 약속으로 고정한다. [Source: _bmad-output/planning-artifacts/prd.md#FR-006-Coupon-Adjusted-Timing-Warning-Alerts-and-Exact-End-Behavior]
- `Session Timing Policy` minimum baseline은 T-5 경고, T=0 exact-end, 승인된 extension과 로그 없는 재개 금지를 요구한다. [Source: _bmad-output/planning-artifacts/prd.md#Named-Policy-References]
- UX는 종료 전후 안내를 고객 안전 문구로 유지하고, warning은 보조 정보이며 post-end는 결과 상태처럼 읽혀야 한다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#구속되는-UX-요구사항] [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Journey-Patterns]
- `session-manifest/v1`은 timing snapshot과 `postEnd`를 함께 유지하며, Rust host만 durable truth를 수정할 수 있다. [Source: docs/contracts/session-manifest.md#필드-규칙]

### 범위 조정 메모

- epics의 Story 2.5 문구는 "Story 3.1이 explicit post-end state를 소유한다"는 전제를 포함한다.
- 그러나 현재 repo에서는 Story 3.1이 이미 완료 문맥으로 존재하므로, dev agent는 그 전제를 그대로 따라가며 `Export Waiting` / `Completed` / `Phone Required`를 숨기거나 늦추면 안 된다.
- 이번 스토리의 안전한 해석은 다음과 같다.
  - explicit post-end truth가 아직 없을 때만 중립 bridge copy 허용
  - explicit post-end truth가 생긴 뒤에는 stronger state 유지
  - Story 3.1/3.2/3.3이 만든 taxonomy와 guidance를 되돌리지 않음

### 선행 의존성과 현재 기준선

- 직접 이전 스토리는 Story 2.4다. 이 스토리가 timing snapshot, warning/end phase, timing cue baseline을 도입했다. [Source: _bmad-output/implementation-artifacts/2-4-조정된-종료-시각-표시와-경고-종료-알림.md#이전-스토리-인텔리전스]
- 현재 코드에는 이미 다음 기준선이 있다.
  - `src-tauri/src/timing/mod.rs`: `warningTriggeredAt`, `endedTriggeredAt`, `captureAllowed`, `timing-events.log`, operator audit
  - `src-tauri/src/handoff/mod.rs`: ended 이후 explicit `postEnd` 평가와 durable manifest 반영
  - `src/booth-shell/selectors/customerStatusCopy.ts`: warning/post-end copy projection
  - `src/booth-shell/screens/ReadinessScreen.tsx`: timing panel, handoff-ready, phone-required, explicit post-end 분기
  - `src/session-domain/state/session-provider.tsx`: stronger post-end state 보존과 stale readiness 완화
- 따라서 이번 스토리는 새 도메인을 만드는 작업보다 기존 경계의 중복 계산과 회귀 가능성을 줄이는 hardening 작업에 가깝다.

### 현재 워크스페이스 상태

- `src/timing-policy/components/SessionTimingPanel.tsx`는 `useEffectEvent` 기반 1초 countdown 보간을 사용하지만, 종료 truth 자체는 host timing snapshot을 소비한다.
- `src/timing-policy/audio.ts`는 warning/ended cue를 Web Audio로 재생하므로, session 재구독이나 rerender 때 중복 재생되지 않게 제어해야 한다.
- `src/capture-adapter/services/capture-runtime.ts`는 Tauri `listen`과 polling을 함께 사용하고 있어, 늦게 도착한 weaker update가 stronger post-end 상태를 덮지 못하도록 현재 guardrail을 유지해야 한다.
- `src-tauri/src/handoff/mod.rs`는 ended 이후 latest capture render status를 보고 `export-waiting`, `completed`, `phone-required`를 고른다. Story 2.5는 이 post-end evaluator를 다시 설계하는 것이 아니라 timing cue와 exact-end boundary가 이 흐름과 제품적으로 자연스럽게 맞물리게 만드는 것이 목적이다.

### 이전 스토리 및 인접 스토리 인텔리전스

- Story 2.4는 host-owned timing truth, same-session threshold crossing, one-shot cue 성격을 이미 강조했다. 이번 스토리는 그 기준을 "제품 신뢰성과 감사 가능성" 관점에서 더 단단히 잠가야 한다. [Source: _bmad-output/implementation-artifacts/2-4-조정된-종료-시각-표시와-경고-종료-알림.md#구현-가드레일]
- Story 3.1은 exact-end 이후 explicit post-end truth를 단일 결과로 수렴시키고 stronger post-end guidance를 보존하는 guardrail을 정리했다. 이번 스토리는 그 흐름을 약화시키지 않고 exact-end bridge copy와 cue timing만 보강해야 한다. [Source: _bmad-output/implementation-artifacts/3-1-종료-직후-명시적-사후-상태-진입.md#구현-가드레일]
- 최근 5개 git commit은 dedicated renderer rollout과 preview diagnostics 중심이며, Story 2.5와 직접 맞닿은 timing/post-end 수정 패턴은 commit log보다 현재 workspace와 implementation artifact 문서에 더 많이 남아 있다.

### 구현 가드레일

- React가 로컬 시간 계산만으로 warning/end truth를 판정하면 안 된다. 표시용 countdown은 가능하지만 authoritative phase와 capture 가능 여부는 host snapshot이 소유해야 한다.
- warning/end 사운드와 배지는 same-session one-shot이어야 한다. rerender, polling, listener 재등록, route 재진입 때문에 다시 울리면 customer trust를 해친다.
- exact-end 이후 capture block은 UI disabled 상태만으로 끝나면 안 된다. host command가 stale client input을 직접 거절해야 한다.
- explicit post-end truth가 이미 생긴 뒤에는 generic `ended` 또는 weaker `export-waiting` snapshot이 stronger `completed`/`phone-required` 상태를 덮으면 안 된다.
- 새 route, 새 lifecycle taxonomy, 새 durable schema version을 도입하지 않는다. `session-manifest/v1`, `session-timing/v1`, `capture-readiness/v1`을 그대로 확장/정렬하는 방향이 안전하다.
- operator extension UI, customer-facing 시간 연장 제어, 새로운 handoff flow는 이번 스토리 범위가 아니다. extension은 audit placeholder와 host-owned hook 수준만 유지한다.

### 최신 기술 확인 메모

- 현재 로컬 기준선은 `react 19.2.4`, `react-router-dom 7.13.1`, `@tauri-apps/api 2.10.1`, `zod 4.3.6`, Rust `tauri 2.10.3`, `tauri-build 2.5.6`이다. 새 의존성 추가보다 이 기준선 안에서 timing/post-end hardening을 마무리하는 편이 안전하다. [Source: package.json] [Source: src-tauri/Cargo.toml]
- React 공식 문서는 `useEffectEvent`를 event/listener 안에서 최신 state를 읽되 effect 재구독을 줄이는 패턴으로 안내한다. timing countdown과 cue trigger cleanup에 적합하다. [Source: https://react.dev/reference/react/useEffectEvent]
- Tauri 2 공식 문서는 frontend listener를 등록한 뒤 scope 종료 시 `unlisten`으로 정리해야 한다고 안내한다. SPA에서 timing/readiness listener 중복을 막는 기준으로 그대로 적용 가능하다. [Source: https://v2.tauri.app/develop/_sections/frontend-listen/]
- Tauri 2 공식 문서는 ordered streaming 성격의 데이터에는 channel을 고려하라고 설명한다. warning/end를 초당 스트리밍으로 확장할 필요가 생기면 event 남용보다 channel 적합성을 먼저 검토해야 한다. 이것은 공식 문서를 바탕으로 한 구현 추론이다. [Source: https://v2.tauri.app/develop/calling-frontend/]
- Tauri 2 공식 문서는 `mockIPC(..., { shouldMockEvents: true })`를 통해 Rust emit 이벤트를 테스트에서 시뮬레이션할 수 있다고 안내한다. warning/end 회귀 테스트 보강에 직접 활용 가능하다. [Source: https://v2.tauri.app/develop/tests/mocking/]
- Zod 공식 문서는 Zod 4를 stable한 TypeScript-first validation library로 소개한다. timing/postEnd contract는 ad-hoc parsing 대신 현재 shared schema family 확장을 유지하는 편이 맞다. [Source: https://zod.dev/]

### 테스트 요구사항

- 최소 필수 테스트 범위:
  - 5분 warning cue와 exact-end cue가 active session에서 정확히 한 번만 발화한다.
  - exact-end 이후 stale polling/event나 route 재진입으로 capture action이 다시 활성화되지 않는다.
  - stronger explicit post-end state가 weaker ended/export-waiting projection에 의해 되돌아가지 않는다.
  - timing-events.log와 operator audit가 warning/end/post-end 분석에 필요한 기준 필드를 남긴다.
  - customer copy에 scheduler, policy, timeout, diagnostics 같은 내부 용어가 새로 노출되지 않는다.
  - approved extension audit ref가 없는 상태에서는 exact-end 이후 captureAllowed가 false로 유지된다.

### 금지사항 / 안티패턴

- warning/end 시각을 프런트 로컬 시계만으로 authoritative truth처럼 계산하는 것 금지
- listener 재구독, polling refresh, rerender 때문에 경고음/종료음이 여러 번 재생되는 것 금지
- ended 이후 UI 버튼만 숨기고 host가 실제 `requestCapture`를 허용하는 상태로 남는 것 금지
