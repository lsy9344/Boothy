# Story 1.3: 승인된 프리셋 카탈로그 표시와 활성 프리셋 선택

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

booth customer로서,
간단한 카탈로그에서 승인된 프리셋 하나를 선택하고 싶다.
그래서 세션 시작 시점과 세션 도중 모두 어떤 룩이 적용되는지 이해하고 필요할 때 바로 바꿀 수 있다.

## Acceptance Criteria

1. 활성 세션이 있는 상태에서 고객이 프리셋 선택 화면에 도달하면, 부스는 고객용 이름과 대표 preview tile 또는 sample cut을 가진 승인된 published preset만 1~6개 보여줘야 한다. 또한 직접 편집 제어와 darktable 용어는 노출되면 안 된다.
2. 고객이 프리셋 선택 화면에서 하나의 프리셋을 선택하거나 다른 프리셋으로 바꾸면, 그 프리셋이 세션의 active preset이 되어야 한다. 또한 active preset identity와 published version은 이후 capture가 참조할 수 있도록 저장되어야 하며, 이미 저장된 capture binding은 바뀌지 않아야 한다.

## Dev Notes

### 스토리 범위와 목적

- 이 스토리는 Story 1.2가 만든 활성 세션 위에서, 고객이 approved published preset을 선택하고 그 선택을 durable session truth에 연결하는 단계다.
- 목표는 "고객이 이해할 수 있는 간단한 preset catalog"와 "세션 중 언제든 바꿀 수 있는 active preset binding"을 만드는 것이다.
- 이 스토리는 preset authoring, preset publication, preview render, final export, operator diagnostics를 구현하는 자리가 아니다.

### 의존성과 선행 조건

- 직접 의존:
  - Story 1.1: 프로젝트 골격, `/booth` route, typed adapter/service seam, `shared-contracts` 경계
  - Story 1.2: active session 생성, booth alias와 durable `sessionId` 분리, session-scoped filesystem root와 초기 `session.json` manifest
- 후속 영향:
  - Story 1.4는 여기서 저장한 active preset binding을 읽어 readiness/capture 가능 여부와 화면 문맥에 반영해야 한다.
  - Story 1.5는 active preset name / version을 capture acknowledgement 및 truthful preview waiting 문맥에 계속 노출해야 한다.
- Story 1.2 story file이 이제 존재하므로, Story 1.3은 그 문서가 정의한 `session.json` v1 기준선과 `activePreset` placeholder 방향을 우선 재사용해야 한다. 같은 세션 필드를 다른 이름으로 병렬 생성하지 말고, Story 1.2의 `start_session` 결과와 같은 booth flow 안에서 preset selection을 이어 붙인다.

### 현재 워크스페이스 상태

- 현재 저장소에는 계획 문서와 Story 1.1 / 1.2 / 1.4 context 문서는 있지만, 실제 앱 스캐폴드는 아직 존재하지 않는다.
- 확인 결과 `package.json`, `pnpm-lock.yaml`, `src/`, `src-tauri/`는 아직 생성되지 않았다.
- Git 저장소도 초기화되어 있지 않아 최근 commit 패턴이나 실제 구현 learnings를 추적할 수 없다.
- 구현은 사실상 Story 1.1이 먼저 적용된 이후의 목표 구조를 기준으로 진행되어야 한다.

### 구현 가드레일

- customer booth는 approved published preset만 보여줘야 한다. draft, validated, approved-but-unpublished preset은 고객 카탈로그에 나오면 안 된다. [Source: _bmad-output/planning-artifacts/epics.md#Story 1.3: 승인된 프리셋 카탈로그 표시와 활성 프리셋 선택] [Source: _bmad-output/planning-artifacts/architecture.md#Core Architectural Decisions]
- booth runtime의 preset truth는 이름 문자열이 아니라 published artifact다. customer UI는 이름과 대표 이미지로 보지만, host/session truth는 `presetId + publishedVersion` 같은 stable identity를 기록해야 한다. [Source: refactoring/2026-03-15-boothy-darktable-agent-foundation.md#9.2 preset artifact 원칙] [Source: reference/darktable/README.md#에이전트가 알아야 할 핵심 포인트]
- active preset binding은 SQLite나 React local state의 장기 진실이 아니라 session-scoped filesystem root의 `session.json` manifest에 남아야 한다. [Source: _bmad-output/planning-artifacts/epics.md#Additional Requirements] [Source: _bmad-output/planning-artifacts/architecture.md#Data Architecture]
- booth UI는 darktable 용어, XMP, style, module 명, authoring internal control, raw library 개념을 직접 노출하면 안 된다. [Source: _bmad-output/planning-artifacts/epics.md#Additional Requirements] [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Reading Guide] [Source: _bmad-output/planning-artifacts/architecture.md#Darktable Capability Scope]
- React component는 직접 `invoke`를 호출하지 말고 typed adapter/service를 거쳐야 한다. preset selection도 동일한 boundary rule을 따라야 한다. [Source: _bmad-output/planning-artifacts/epics.md#Additional Requirements] [Source: _bmad-output/planning-artifacts/architecture.md#Architectural Boundaries]
- preset selection은 현재 세션의 active preset binding을 설정하거나 갱신하는 단계다. 이 시점에 preview render path, final export path, current-session asset mutation을 섞지 말 것. 변경은 이후 촬영부터 반영되지만 이미 저장된 capture asset은 재바인딩하지 않는다. [Source: refactoring/2026-03-15-boothy-darktable-agent-foundation.md#9. 프리셋 기능의 제품 내 위치] [Source: _bmad-output/planning-artifacts/architecture.md#Data Architecture]

### 아키텍처 준수사항

- 제품은 하나의 packaged Tauri application 안에서 booth, operator, authoring surface를 capability-gated로 분리한다. 이 story의 UI 작업은 booth surface에만 속해야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Core Architectural Decisions]
- FR-001 / FR-002 구현 영역은 `src/booth-shell/`, `src/preset-catalog/`, `src-tauri/src/commands/session_commands.rs`, `src-tauri/src/preset/`로 매핑되어 있다. [Source: _bmad-output/planning-artifacts/architecture.md#Requirements to Structure Mapping]
- session folder는 `session.json`, `captures/originals/`, `renders/previews/`, `renders/finals/`, `handoff/` 같은 구조를 전제로 한다. Story 1.3은 최소한 `session.json`에 preset binding을 넣는 방향과 충돌하지 않아야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Data Architecture]
- preset/session separation 규칙상 preset-authoring은 active booth session data를 직접 수정하지 않는다. booth session은 published preset artifact를 reference만 해야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Data Architecture]
- host authority 규칙상 session file mutation과 preset bundle validation은 Rust host가 책임져야 한다. UI가 파일 경로나 artifact validity를 임의 판단하면 안 된다. [Source: _bmad-output/planning-artifacts/architecture.md#Security & Operational Boundaries]

### 데이터 및 상태 모델 메모

- customer-facing card가 표시하는 값과 durable binding에 필요한 값은 구분한다.
  - customer-facing: `displayName`, preview tile 또는 sample cut, selected state
  - durable binding: `presetId`, `publishedVersion`, optional preview/final profile reference
- catalog item은 published preset bundle에서 파생된 booth-safe read model이어야 하며, darktable module parameter나 internal file layout을 UI에 흘리지 않는다.
- active session은 한 시점에 하나의 active preset만 가진다.
- active preset은 세션 중 언제든 다시 선택될 수 있지만, 변경은 그 시점 이후 촬영부터 반영되고 이미 저장된 capture asset은 재바인딩하지 않는다.

### UX 요구사항

- preset card는 큰 터치 영역, 높은 대비, 대표 이미지, 룩 이름, 선택 강조를 포함해야 한다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Custom Components]
- booth UI는 Brutal Core 방향과 high-contrast, touch-friendly 원칙을 유지해야 한다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Design Direction Decision] [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Responsive Design & Accessibility]
- 고객 기본 문구는 low-density, plain-language 원칙을 따라야 한다. preset 선택 화면에서도 "기술 설명"보다 "어떤 룩인지 바로 고를 수 있는 안내"가 우선이다. [Source: _bmad-output/planning-artifacts/prd.md#Non-Functional Requirements] [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Reading Guide]
- 운영자/저작 진입점은 고객 기본 흐름에서 숨겨져 있어야 하므로 preset selection screen에 internal action entry를 두면 안 된다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Reading Guide]
