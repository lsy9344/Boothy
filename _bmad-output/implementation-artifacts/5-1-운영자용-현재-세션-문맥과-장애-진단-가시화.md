# Story 5.1: 운영자용 현재 세션 문맥과 장애 진단 가시화

Status: done

## Story

remote operator로서,
현재 세션 문맥과 blocked-state 진단 정보를 보고 싶다.
그래서 부스가 capture, preview/render, completion 중 어디에서 막혔는지 추측 없이 이해할 수 있다.

## Acceptance Criteria

1. booth session이 active 또는 blocked 상태일 때 운영자가 operator console을 열면, console은 현재 session identity, timing state, recent failure context, normalized blocked-state category를 보여줘야 한다. 또한 화면은 capture-side blockage와 preview/render/completion blockage를 구분해서 보여줘야 한다.
2. operator console이 blocked session의 진단 정보를 렌더링할 때, UI는 raw helper output 대신 operator-safe diagnostic detail을 사용해야 한다. 또한 customer-facing booth copy와 기본 고객 흐름은 영향을 받지 않아야 한다.

## Dev Notes

### 스토리 범위와 목적

- 이 스토리는 Epic 5 / FR-009의 첫 가치 조각이다. 핵심은 "복구를 수행하는 것"이 아니라, 운영자가 현재 세션과 막힌 경계를 빠르게 이해할 수 있는 읽기 전용 진단 시야를 여는 것이다.
- Story 5.2가 정책 기반 복구 액션을 실행하는 단계라면, 5.1은 그 전제인 blocked-state classification과 operator-safe context projection을 고정하는 단계다.
- Story 5.3의 본격 감사 로그/개입 이력보다 앞서, 5.1은 지금 이미 존재하는 session manifest, timing/post-end truth, diagnostics 디렉터리를 바탕으로 "현재 무엇이 막혔는가"를 안전하게 보여주는 범위에 집중해야 한다.

### 스토리 기반 요구사항

- 운영자는 current session context, timing state, recent failure context, allowed action set을 이해할 수 있어야 하며, blocked state는 capture / preview-render / completion 경계로 구분돼야 한다. [Source: _bmad-output/planning-artifacts/prd.md#FR-009 Operational Safety and Recovery]
- `Operator Recovery Policy` baseline은 blocked states를 capture, preview or render, post-end recovery categories로 정규화한 뒤 operator tools에 노출하도록 요구한다. [Source: _bmad-output/planning-artifacts/prd.md#Named Policy References]
- operator journey는 current session visibility, capture-state and render-state fault diagnosis, lifecycle/queue/intervention review를 포함해야 한다. [Source: _bmad-output/planning-artifacts/prd.md#Mode B: Operator Recovery Surface]
- blocked state 진단은 raw helper output 대신 operator-safe detail을 사용해야 하고, customer-facing booth copy는 영향을 받지 않아야 한다. [Source: _bmad-output/planning-artifacts/epics.md#Story 5.1: 운영자용 현재 세션 문맥과 장애 진단 가시화]

### 선행 의존성과 구현 순서

- 직접 선행 문맥은 Story 1.5, Story 2.4, Story 3.1, Story 3.2, Story 3.3이다.
- Story 1.5는 capture success와 preview readiness를 분리하고 latest capture / renderStatus contract를 고정했다. 5.1은 이 capture truth를 operator 분류의 입력으로 재사용해야 한다.
- Story 2.4는 host-owned timing truth와 warning/exact-end 차단을 정리했다. 5.1의 timing/post-end blockage는 이 host timing snapshot에서 읽어야 한다.
- Story 3.1 ~ 3.3은 explicit post-end truth(`export-waiting`, `completed`, `phone-required`)와 handoff/phone-required 보호 안내를 고정했다. 5.1은 이 post-end truth를 operator 관점의 completion blockage로 재해석해야 한다.
- Story 5.2와 5.3은 아직 backlog다. 따라서 5.1은 action execution이나 full audit explorer를 선구현하지 말고, 읽기 전용 summary와 stable classification seam을 먼저 세워야 한다.

### 현재 워크스페이스 상태

- `src/operator-console/screens/OperatorSummaryScreen.tsx`는 아직 placeholder이며 실제 session context, diagnostics, recovery 분류를 보여주지 않는다.
- `src/app/routes.tsx`에는 `/operator` route와 `SurfaceAccessGuard surface=\"operator\"`가 이미 있다. 운영자 surface 자체는 준비돼 있지만 실제 도메인 상태와 UI가 비어 있다.
- `src/app/providers/app-providers.tsx`는 capability, preset-authoring, booth `SessionProvider`만 연결하고 있으며 operator domain provider는 아직 없다.
- `src-tauri/src/commands/mod.rs`에는 `capture_commands`, `preset_commands`, `runtime_commands`, `session_commands`만 등록돼 있고 operator/diagnostics command module이 아직 없다.
- `src/session-domain/state/session-provider.tsx`와 `src/capture-adapter/services/capture-runtime.ts`는 booth flow를 위한 same-session guard, event/poll merge, host error normalization을 이미 갖고 있다. 그러나 operator surface는 booth flow local state를 재사용하기보다 host read-model을 별도로 가져가는 편이 안전하다.
- `src-tauri/src/session/session_paths.rs`, `src-tauri/src/timing/mod.rs`, `src-tauri/src/handoff/mod.rs`는 session `diagnostics/` 디렉터리와 `timing-events.log` 등 운영용 단서를 이미 만들고 있다. 5.1은 이 현재 기준선을 operator-safe summary로 읽는 첫 단계가 될 수 있다.

### 구현 가드레일

- operator diagnostics는 booth `SessionProvider`의 화면 편의 state를 복사한 값이 아니라, host가 만든 정규화된 summary projection이어야 한다.
- blocked-state category는 operator action policy의 입력이 되므로 UI에서 ad-hoc 분기 문자열을 만들지 말고 shared contract enum으로 고정하는 편이 안전하다.
- raw helper stderr, filesystem path, serde parse error, darktable 내부 용어, Rust enum 이름을 그대로 화면에 노출하면 안 된다. operator-safe detail은 충분히 구체적이어야 하지만 내부 구현 원문이어서는 안 된다.
- customer booth copy와 operator diagnostics는 같은 normalized truth에서 갈라져야 한다. operator 기능을 추가하면서 booth customer wording이나 화면 위계를 바꾸면 안 된다.
- active session이 없을 때도 operator surface는 크래시하거나 이전 세션 정보를 붙잡으면 안 된다. empty state 또는 typed no-session summary가 필요하다.
- 이번 스토리에서는 recovery action 실행, time extension, restart button, audit explorer를 미리 넣지 않는다. 그런 행동은 Story 5.2/5.3 범위다.

### UX 구현 요구사항

- operator surface는 customer booth UI와 달리 더 높은 정보 밀도를 가질 수 있지만, 여전히 raw helper output dump가 아니라 "행동 가능한 운영자 진단"이어야 한다.
- 현재 세션 identity는 booth alias, sessionId, active preset, lifecycle/timing/post-end truth를 함께 보여 주되, 불필요한 교차 세션 참조나 과도한 PII는 노출하지 않는 편이 안전하다.
- blocked-state category는 한눈에 읽히는 badge/card 위계가 필요하다. 특히 capture-side blockage와 preview/render/completion blockage를 같은 문단에 섞지 말고 구획을 분리해야 한다.
- customer-facing booth copy는 여전히 plain-language 보호 카피를 유지해야 하며, operator diagnostics terminology가 booth 화면에 새어 나오면 안 된다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Reading Guide]
- `/operator` surface는 768px 이상 운영자 화면 범위를 지원해야 하고, booth 기본 흐름에서는 계속 숨겨져 있어야 한다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Responsive Design & Accessibility]

### 최신 기술 확인 메모

- 현재 로컬 기준선은 `react 19.2.4`, `react-router-dom 7.13.1`, `@tauri-apps/api 2.10.1`, `zod 4.3.6`, `tauri 2.10.3`이다. 이번 스토리는 새 프레임워크보다 existing typed boundary 위에 operator diagnostics read-model을 추가하는 편이 안전하다. [Source: package.json] [Source: src-tauri/Cargo.toml]
- Tauri v2 문서는 privileged backend work를 `#[tauri::command]` 경계로 노출하는 방식을 전제로 한다. 5.1의 operator summary는 one-shot diagnostics query이므로 command 중심 구현이 적합하다. [Source: https://v2.tauri.app/develop/calling-rust/]
- Tauri v2 문서는 event system이 low latency/high throughput streaming용이 아니고, ordered streaming에는 channels가 더 적합하다고 설명한다. 5.1은 우선 command + explicit refresh/poll로 시작하고, 지속 스트리밍이 필요해질 때 channel 전환을 검토하는 편이 안전하다. 이것은 공식 문서와 현재 요구사항을 바탕으로 한 구현 추론이다. [Source: https://v2.tauri.app/develop/calling-frontend/]
- Tauri capability 문서는 capability가 window/webview에 붙는 권한 모델임을 설명한다. operator diagnostics command와 UI는 기존 operator surface capability를 재사용해야 하며 booth window를 통해 우회 노출되면 안 된다. [Source: https://v2.tauri.app/reference/acl/capability/]
- React 공식 문서는 `useEffectEvent`가 effect/listener 안에서 최신 값을 읽는 용도라고 설명한다. operator refresh/poll 또는 event bridge를 붙일 때 stale closure를 줄이는 데 유용하지만 dependency shortcut 용도로 오용하면 안 된다. [Source: https://react.dev/reference/react/useEffectEvent]
- Zod 공식 사이트는 Zod 4가 stable이며 TypeScript-first validation library라고 명시한다. operator summary도 ad-hoc parsing보다 shared schema family에 넣는 편이 안전하다. [Source: https://zod.dev/]

  - capture-side blockage와 preview/render/completion blockage가 서로 다른 category로 분류된다.
  - raw helper output, filesystem path, internal Rust error text가 operator UI에 직접 나타나지 않는다.
  - active session이 없을 때 safe empty state 또는 typed no-session summary가 나온다.
  - `/operator` route는 capability/auth 없이는 계속 접근 불가이고, booth 기본 UI에는 operator entry가 나타나지 않는다.
  - stale/foreign response 또는 malformed diagnostics snapshot이 현재 operator 화면을 오염시키지 않는다.

### 금지사항 / 안티패턴

- `OperatorSummaryScreen` 안에서 직접 `invoke`를 호출하는 것 금지
- booth `SessionProvider` state를 operator truth처럼 그대로 재사용하는 것 금지
- raw helper stderr/path, render worker text, filesystem path를 operator-safe detail이라고 주장하며 그대로 노출하는 것 금지
- blocked-state category를 JSX 안에서 임의 문자열 비교로만 계산하는 것 금지
- Story 5.2 범위인 recovery action button과 Story 5.3 범위인 full audit explorer를 이번 스토리 안에서 과도하게 선구현하는 것 금지
- operator diagnostics 추가 때문에 booth customer copy, route, capability visibility가 흔들리는 것 금지
