## Story

authorized preset manager로서,
문제가 있는 게시 프리셋을 이전 승인 버전으로 되돌리고 싶다.
그래서 활성 세션을 깨지 않으면서 미래 세션용 카탈로그를 안전하게 복구할 수 있다.

## Acceptance Criteria

1. 동일한 preset identity에 대해 여러 개의 승인된 게시 버전이 존재할 때 관리자가 rollback target을 고르면, 시스템은 선택된 이전 승인 버전을 미래 세션용 catalog version으로 만든다. 또한 이미 진행 중인 활성 세션은 현재 바인딩된 preset version을 그대로 유지해야 한다.
2. publication 또는 rollback이 완료되면 시스템은 preset identity, version, action type, timestamp, actor를 audit history에 기록해야 한다. 또한 branch-visible catalog state는 내부적으로 일관된 상태를 유지해야 한다.

## Dev Notes

### 스토리 범위와 목적

- 이 스토리는 Epic 4 / FR-008의 마지막 가치 조각이다. 목적은 "게시된 프리셋을 지우는 것"이 아니라, 불변 published artifact는 그대로 두고 미래 세션용 catalog pointer만 안전하게 되돌리는 것이다.
- 이 스토리는 Epic 6의 지점 배포/앱 빌드 rollback을 다루지 않는다. 여기서의 rollback은 동일 앱/동일 branch 안에서 승인된 preset stack 중 어떤 published version이 booth catalog에 노출되는지를 되돌리는 작업이다.
- 이 스토리는 세션 중 프리셋 변경 기능을 다시 만드는 작업도 아니다. booth customer의 현재 세션 binding은 그대로 두고, authoring/operator 권한 안에서 future-session catalog만 조정해야 한다.

### 스토리 기반 요구사항

- preset publication과 rollback은 미래 세션에만 영향을 주고 active session data 또는 active preset binding을 직접 바꾸면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#Rollback-and-Future-Session-Safety]
- 승인된 booth preset만 customer catalog에 보여야 하며, 각 preset은 stable identity와 published version을 가진 immutable artifact bundle이어야 한다. [Source: _bmad-output/planning-artifacts/prd.md#Published-Preset-Artifact-Model]
- FR-008은 draft -> validated -> approved -> published lifecycle과 rollback capability를 함께 요구한다. 4.4는 4.3의 immutable publish baseline을 재사용해야지 별도 publish path를 새로 만들면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#FR-008-Authorized-Preset-Authoring,-Approval,-and-Publication]
- publication/rollback action은 audit history에 identity, version, action type, timestamp, actor를 남겨야 한다. [Source: _bmad-output/planning-artifacts/epics.md#Story-4.4:-미래-세션-대상-롤백과-카탈로그-버전-관리]

### 선행 의존성과 구현 순서

- 논리적 선행 순서는 Story 4.1 -> 4.2 -> 4.3 -> 4.4다.
- 4.4는 4.3이 만든 immutable published bundle contract 위에 catalog activation layer를 얹는 단계다.
- 현재 코드베이스에는 customer-facing published catalog reader와 bundle validator가 이미 있으므로, 4.4는 이 기반을 확장해야 한다.
- 만약 4.1~4.3이 아직 코드로 완성되지 않았더라도, 4.4 구현 중 임시 publication path를 새로 만드는 대신 4.3 acceptance criteria를 만족하는 fixture/baseline을 기준으로 rollback 경계를 세워야 한다.

### 현재 워크스페이스 상태

- `src/preset-authoring/screens/PresetLibraryScreen.tsx`는 아직 placeholder이며, 실제 버전 관리나 rollback UI는 없다.
- `/authoring` route는 이미 `SurfaceAccessGuard`와 capability service로 보호되고 있고, `src-tauri/capabilities/authoring-window.json`도 future authoring surface용 placeholder만 있다.
- `src-tauri/src/preset/preset_catalog.rs`는 `preset-catalog/published/{presetId}/{publishedVersion}/bundle.json`을 스캔한 뒤, preset identity마다 lexicographically 가장 큰 `published_version`을 현재 customer catalog version으로 간주한다.
- 이 최신-version heuristic은 rollback과 충돌한다. 이전 승인 버전으로 되돌리려면 newer bundle을 삭제해야만 하는 구조가 되는데, 그것은 immutability를 깨는 잘못된 접근이다.
- `src-tauri/src/preset/preset_bundle.rs`는 `published-preset-bundle/v1`, `lifecycleStatus = published`, `boothStatus = booth-safe`, preview asset path 안전성 등을 검증한다. 즉, immutable bundle reader는 이미 강한 기초가 있다.
- `src-tauri/src/session/session_repository.rs`와 `docs/contracts/session-manifest.md`는 active session의 `activePreset` binding을 manifest에 저장하고, capture record는 촬영 시점 preset version을 보존한다. 이 경계가 active session non-mutation의 핵심 기반이다.
- 현재 코드베이스에는 preset publication/rollback audit용 SQLite 또는 동등한 dedicated audit repository가 아직 없다. audit requirement는 이번 스토리에서 host-owned seam으로 명시적으로 시작해야 한다.
- 아키텍처는 `docs/contracts/preset-bundle.md`를 기대하지만 현재 repo에는 `docs/contracts/session-manifest.md`만 존재한다. 4.4에서 preset bundle/catalog state 문서를 추가하거나, 최소한 authoring contract reference를 문서화할 필요가 있다.

### 구현 가드레일

- rollback을 구현하기 위해 newer published bundle directory를 삭제, rename, overwrite, backdate 하면 안 된다. published bundle은 계속 immutable이어야 한다.
- 이번 스토리 이후에는 "preset identity별 최신 `publishedVersion`이 곧 live catalog version"이라는 가정을 유지하면 안 된다. live catalog state는 별도의 host-owned pointer/revision이 결정해야 한다.
- future-session-only를 엄격히 지키려면, active session이 참조하는 catalog baseline을 session boundary 안에 고정해야 한다. 최소한 이미 바인딩된 `manifest.activePreset`은 rollback 이후에도 절대 바뀌면 안 된다.
- booth customer surface는 여전히 승인된 게시 프리셋만 보여야 하며, draft/validated/approved-but-not-live 버전 목록이나 rollback affordance를 고객에게 노출하면 안 된다.
- rollback 거절은 "target version 없음", "승인/호환성 기준 미달", "이미 활성 버전", "audit 저장 실패"처럼 host가 결정해야 한다. JSX에서 optimistic local mutation 후 되돌리는 방식은 피하는 편이 안전하다.
- audit는 console log나 scattered text file로 흩어지면 안 된다. 추후 SQLite로 옮길 수 있는 단일 repository seam을 두고, action schema를 먼저 고정해야 한다.

### UX 구현 요구사항

- customer flow는 여전히 "승인된 published preset 선택"만 보여야 하고 authoring, rollback, darktable, XMP 같은 내부 용어를 드러내면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#Customer-Preset-Experience-Baseline]

- rollback confirmation은 "활성 세션은 바뀌지 않고, 새로 시작하는 세션부터 적용된다"는 제품 약속을 분명히 보여 줘야 한다.
- rejection copy는 authorized user에게 행동 가능한 이유를 줘야 하지만 raw path, serde parse error, internal enum 이름을 그대로 보여 주는 것은 피하는 편이 좋다.

### 최신 기술 확인 메모

- 현재 로컬 기준선은 `react 19.2.4`, `react-router-dom 7.13.1`, `@tauri-apps/api 2.10.1`, `zod 4.3.6`, `tauri 2.10.3`이다. 이번 스토리는 새 프레임워크 도입보다 기존 typed boundary 안에서 catalog state와 rollback command를 추가하는 편이 안전하다. [Source: package.json] [Source: src-tauri/Cargo.toml]
- Tauri v2 command 문서는 privileged backend work를 `#[tauri::command]` 경계로 두고 `invoke_handler`에 등록하는 패턴을 전제로 한다. rollback은 continuous stream이 아니라 low-frequency privileged mutation이므로 command 중심 구현이 맞다. [Source: https://v2.tauri.app/develop/calling-rust/]
- Tauri capability 문서는 window별 capability files를 통해 surface 권한을 분리하는 구조를 설명한다. authoring rollback action도 기존 authoring capability/profile gating을 재사용해야 한다. [Source: https://v2.tauri.app/reference/acl/capability/]
- Tauri frontend/event 문서는 high-throughput ordered stream에는 channels가 더 적합하다고 설명한다. rollback은 one-shot mutation이므로 기본 command + explicit refresh가 우선이고, 실시간 history stream을 새로 만들 필요는 낮다. 이것은 공식 문서와 현재 요구사항을 바탕으로 한 구현 추론이다. [Source: https://v2.tauri.app/develop/calling-frontend/]
- React 공식 문서에서 `useEffectEvent`는 effect-fired logic에서 최신 값을 읽는 용도이지 stable callback identity 대체재가 아니다. authoring confirmation/modal side effect에 쓰더라도 dependency shortcut 용도로 오용하면 안 된다. [Source: https://react.dev/reference/react/useEffectEvent]
- Zod 공식 사이트는 Zod 4가 stable이라고 명시한다. rollback/catalog state 계약도 ad-hoc parsing 대신 shared schema family 안에 넣는 편이 안전하다. [Source: https://zod.dev/]

### 테스트 요구사항

- 최소 필수 테스트 범위:
  - older approved version으로 rollback해도 newer published bundle directory와 `bundle.json`이 그대로 남는다.
  - `load_preset_catalog`가 더 이상 "가장 큰 version"을 자동 선택하지 않고 active catalog pointer를 따른다.
  - rollback 이전에 이미 preset을 바인딩한 active session은 같은 `activePreset`을 유지한다.
  - rollback 이후 새로 시작한 session만 바뀐 catalog version을 본다.
  - invalid/missing/incompatible target rollback은 audit 실패를 포함해 catalog mutation 없이 거절된다.
  - `/authoring` surface는 admin auth + capability 없이는 접근되지 않고, booth UI 어디에도 rollback entry가 생기지 않는다.
  - audit entry에 preset identity, from/to version, action type, actor, timestamp가 빠지지 않는다.

### 금지사항 / 안티패턴
