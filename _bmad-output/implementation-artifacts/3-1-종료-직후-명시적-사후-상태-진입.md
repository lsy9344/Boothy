# Story 3.1: 종료 직후 명시적 사후 상태 진입

## Story

booth customer로서,
촬영이 끝나는 즉시 부스가 분명한 사후 상태로 넘어가길 원한다.
그래서 지금 기다려야 하는지, 마무리됐는지, 도움을 요청해야 하는지를 바로 이해할 수 있다.

## Acceptance Criteria

1. 활성 세션이 조정된 종료 시각에 도달해 더 이상 촬영이 허용되지 않으면, 부스는 `Export Waiting`, `Completed`, `Phone Required` 중 정확히 하나의 명시적 사후 상태로 전환되어야 한다. 또한 고객은 애매한 capture 중간 상태에 남아 있으면 안 된다.
2. 부스가 사후 상태에 들어가면, 고객 화면은 기술 진단어 없이 고객 안전 문구를 사용해야 한다. 또한 다음 행동은 시각적으로 분명해야 한다.

## Dev Notes

### 스토리 범위와 목적

- 이 스토리는 Epic 3 / FR-007의 첫 가치 조각으로, "종료 직후 부스가 어떤 상태인지"를 애매하지 않게 만드는 작업이다.
- 핵심은 종료를 알리는 것 자체가 아니라, exact-end 직후 고객과 시스템이 모두 하나의 explicit post-end truth를 바라보게 하는 것이다.
- Story 2.4가 warning/exact-end와 capture 차단 경계를 다진다면, Story 3.1은 그 직후 `Export Waiting`, `Completed`, `Phone Required` 중 하나로 진입시키는 host-owned 분기점을 만든다.
- Story 3.2와 Story 3.3은 각각 truthful completion copy/variant와 handoff-ready 또는 phone-required 보호 경험을 더 구체화하는 후속 단계다. 이번 스토리에서는 "명시적 진입"이 우선이다.

### 스토리 기반 요구사항

- 종료 직후 product는 `Export Waiting`, `Completed`, `Phone Required` 중 하나의 explicit post-end state로 진입해야 한다. [Source: _bmad-output/planning-artifacts/prd.md#FR-007 Export Waiting, Final Readiness, and Handoff Guidance]
- post-end truth는 PRD의 post-end completion taxonomy를 따라야 하며 branch-local 임의 상태를 발명하면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#Post-End Completion Taxonomy]
- 90% 이상의 세션은 scheduled end 이후 10초 안에 explicit post-end state로 진입해야 한다. [Source: _bmad-output/planning-artifacts/prd.md#NFR-005 Timing, Post-End, and Render Reliability]
- 고객 화면은 기술 진단어 없이 다음 행동을 명확히 보여줘야 하며, 특히 `Phone Required`는 보호 화면이어야 한다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Phone Required-보호-흐름]

### 선행 의존성과 구현 순서

- 직접 선행 스토리는 Story 2.4다. Story 2.4가 adjusted end time, warning, exact-end, capture block을 안정화해야 Story 3.1이 그 직후 explicit post-end state로 넘길 수 있다.
- 다만 현재 `3-1` 생성 시점 기준으로 `2-4`는 `ready-for-dev` 상태이며 아직 `done`이 아니다. 따라서 `3-1` 구현자는 timing 기준선이 막 들어오는 상황을 전제로 작업해야 한다.
- Story 1.5와 Story 2.1~2.3에서 이미 capture truth, current-session rail, deletion guard, forward-only preset change가 정리됐으므로, 3.1은 이 위에 post-end state entry를 얹어야 한다.
- 구현 순서는 안전하게 다음이 좋다.
  - timing truth를 실제 capture readiness 경로에 연결
  - explicit post-end contract와 manifest/state 확장
  - booth UI에서 capture-loop 액션 제거 및 post-end panel 도입
  - 회귀 테스트 추가

### 현재 워크스페이스 상태

- 현재 workspace에는 `src/shared-contracts/schemas/session-timing.ts`, `src/shared-contracts/dto/timing.ts`, `src-tauri/src/timing/mod.rs`가 이미 생겨 있어 Story 2.4 방향의 timing scaffold가 시작된 상태다.
- Rust `build_session_manifest_at()`는 기본 `timing` snapshot을 세션 시작 시 manifest에 기록한다. `sync_session_timing_in_dir()`는 warning/ended 판정과 diagnostics log를 남길 수 있지만, 아직 capture command/readiness 경로에 완전히 연결돼 있지 않다.
- `CaptureReadinessDto`는 이미 `export_waiting`, `completed`, `warning`, `ended` constructor를 가지고 있지만, 현재 `export_waiting`과 `completed`는 `Preparing`/`wait` 기반의 임시 표현이라 explicit post-end state라고 보기 어렵다.
- 프런트엔드 `customerReadinessStateSchema`는 아직 `Export Waiting`과 `Completed`를 customer state로 허용하지 않는다. 대신 `Session Ended`만 있어 exact-end 직후 애매한 상태에 머물 가능성이 남아 있다.
- `CaptureScreen`과 `ReadinessScreen`은 현재 촬영 루프 중심 화면이다. 현재 구조에서는 종료 후에도 "다음 촬영 룩 바꾸기"와 capture action card가 남을 수 있으므로, post-end 전용 표현 분기가 필요하다.
- `SessionDraft`는 여전히 `flowStep: 'capture'` 하나로 booth loop와 post-end를 함께 담고 있다. 3.1은 별도 route를 도입하기보다 이 상태 기계 안에서 explicit post-end projection을 추가하는 편이 현재 구조와 잘 맞는다.

### 구현 가드레일

- exact-end 이후 상태는 host가 소유해야 한다. React가 로컬 clock 또는 route state만으로 `Export Waiting` / `Completed` / `Phone Required`를 판단하면 안 된다.
- 종료 직후에는 반드시 하나의 explicit post-end state만 남아야 한다. `ended`, generic `Preparing`, `Ready`, `warning` 같은 중간 표현이 customer-safe state처럼 계속 노출되면 안 된다.
- `Completed`는 booth-side required work가 실제로 끝났을 때만 써야 한다. "곧 끝날 것 같음"이나 "렌더 재시도 중"은 모두 `Export Waiting` 또는 `Phone Required` 경계에 남겨야 한다.
- 이미 저장된 current-session capture truth는 post-end 분기 때문에 사라지거나 교차 세션으로 섞이면 안 된다.
- 종료 후에는 capture/preset-switch 액션을 고객이 다시 시도하게 만드는 UI를 남기면 안 된다. 다음 행동은 wait / finish / call 중 하나로 좁혀야 한다.
- Story 3.2 / 3.3 범위까지 과도하게 선구현하지 말아야 하지만, 이번 스토리에서 state 명칭과 경계만큼은 후속 스토리가 그대로 쓸 수 있게 고정해야 한다.

### UX 구현 요구사항

- 종료 후 다음 행동은 고객 안전 문구로 명확해야 하고, 기술 원인이나 내부 처리 단계를 설명하듯 쓰면 안 된다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#구속되는-ux-요구사항]
- `Phone Required`는 도움 요청 중심 보호 화면이어야 하며, 연락 방법 하나와 고객이 하지 말아야 할 행동을 짧게 보여줘야 한다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Phone Required-보호-흐름]
- `Preview Waiting`과 `Phone Required`는 단일 행동 원칙을 따르는 별도 재사용 컴포넌트 방향으로 설계되어야 한다. post-end 패널도 같은 위계를 따르는 것이 안전하다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Journey-Patterns]
- 종료 후 상태도 촬영 루프의 연장선이 아니라 결과 상태처럼 읽혀야 한다. "마무리 준비 중", "안내를 확인해 주세요", "도움이 필요해요"는 가능하지만 scheduler/render-engine/policy 같은 내부어는 금지다.

### 최신 기술 확인 메모

- 현재 로컬 기준선은 `react 19.2.4`, `react-router-dom 7.13.1`, `@tauri-apps/api 2.10.1`, `zod 4.3.6`이다. 이번 스토리는 새 의존성보다 기존 기준선 안에서 post-end truth를 연결하는 편이 안전하다. [Source: package.json]
- React 공식 문서는 `useEffectEvent`가 timer/event listener 안에서 최신 render 값을 읽으면서 effect 재구독을 피하는 패턴을 제공한다고 설명한다. post-end sound/threshold listener나 subscription cleanup 로직에 적합하다. [Source: https://react.dev/reference/react/useEffectEvent]
- Tauri 공식 문서는 `listen`이 반환하는 `unlisten`을 SPA에서 scope 종료 시 직접 호출해야 한다고 안내한다. post-end readiness listener도 session 변경/언마운트 때 정리돼야 한다. [Source: https://v2.tauri.app/develop/_sections/frontend-listen/]
- Tauri 공식 문서는 event system이 대용량/저지연 스트리밍용이 아니고, ordered streaming에는 channels가 더 적합하다고 설명한다. exact-end 이후 연속 상태 스트림이 커진다면 channel 전환을 고려할 수 있다. 이것은 공식 문서를 바탕으로 한 구현 추론이다. [Source: https://v2.tauri.app/develop/calling-frontend/]
- Tauri 공식 문서는 `mockIPC(..., { shouldMockEvents: true })`로 Rust emit 이벤트를 테스트에서 시뮬레이션할 수 있다고 안내한다. post-end event regression test에 바로 활용 가능하다. [Source: https://v2.tauri.app/develop/tests/mocking/]
- Zod 공식 문서는 Zod 4를 stable, TypeScript-first validation library로 소개한다. 새 post-end DTO도 기존 shared schema family 안에서 확장하는 편이 맞다. [Source: https://zod.dev/]

### 테스트 요구사항

- 최소 필수 테스트 범위:
  - exact-end 이후 readiness가 `Export Waiting`, `Completed`, `Phone Required` 중 하나로만 나온다.
  - `Export Waiting`과 `Completed`가 더 이상 `Preparing` 같은 generic customer state로 파싱되지 않는다.
  - post-end 상태에 들어간 뒤 capture와 in-session preset switch가 모두 차단된다.
  - late pre-end readiness, foreign session event, stale poll 응답이 현재 post-end state를 pre-end 상태로 되돌리지 못한다.
  - `Phone Required`는 단일 연락 행동 중심의 보호 화면으로 보이고, capture-loop 버튼은 남지 않는다.
  - 종료 직후 post-end state가 10초 budget 안에 기록될 수 있도록 host timestamp/log가 남는다.

### 금지사항 / 안티패턴

- exact-end 이후 `Session Ended`를 최종 customer state처럼 오래 노출하는 것 금지
- `Export Waiting` / `Completed`를 customer-safe explicit state로 만들지 않고 `Preparing` 재사용으로 덮는 것 금지
- UI 버튼만 비활성화하고 host는 여전히 capture나 preset switch를 허용하는 상태로 두는 것 금지
- route 이동 자체를 post-end truth의 기준으로 삼는 것 금지
- Story 3.2 / 3.3 세부 완료/인계 UX를 아직 증명되지 않은 상태에서 거짓 completion copy로 선반영하는 것 금지
