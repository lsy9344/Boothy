# Story 2.1: 현재 세션 사진 레일과 세션 범위 검토

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

booth customer로서,
내 현재 세션의 사진만 검토하고 싶다.
그래서 다른 사람 사진이 섞이지 않는 상태에서 방금 찍은 결과를 안심하고 확인할 수 있다.

## Acceptance Criteria

1. 활성 세션에 성공적으로 저장된 촬영이 하나 이상 있을 때 고객이 검토 레일을 보면, UI는 활성 세션에 상관된 자산만 보여줘야 한다. 또한 현재 세션에서 가장 최신으로 확인 가능한 사진은 레일 안에서 시각적으로 구분되어야 한다.
2. 검토 surface가 렌더링될 때 세션 범위 자산이 로드되면, 이전 세션 또는 다른 세션 자산은 절대 보이면 안 된다. 또한 고객용 UI에는 filesystem, render-engine, diagnostic 용어를 노출하면 안 된다.
3. 현재 세션 검토 데이터가 조회, 갱신, 삭제 흐름에 재사용될 때 booth-safe 자산 해석은 활성 세션 범위로만 한정되어야 한다. 따라서 review, preview, deletion 흐름 전체에서 cross-session asset leakage는 0이어야 한다.

## Dev Notes

### 스토리 범위와 목적

- 이 스토리는 Epic 2에서 FR-005의 첫 번째 가치 조각이다. 핵심은 고객이 "내 현재 세션의 사진만" 보게 하고, 그중 최신으로 확인 가능한 컷을 즉시 이해하게 만드는 것이다.
- 삭제 정책의 완성 자체는 Story 2.2 범위지만, Story 2.1은 review rail과 session-scoped asset 해석을 먼저 안정화해서 이후 삭제 흐름이 같은 진실 경계를 재사용할 수 있게 해야 한다.
- 이 스토리는 새로운 편집 기능이나 별도 갤러리 제품을 만드는 작업이 아니다. booth 메인 흐름 안에서 current-session review confidence를 보강하는 작업이다.

### 선행 의존성과 구현 순서

- 선행 구현은 Story 1.2 ~ 1.5다.
- 특히 Story 1.5에서 이미 다음 기반이 준비되어 있다.
  - source photo persistence와 preview readiness truth 분리
  - `Preview Waiting` 보호 패널
  - 현재 세션 manifest 기반 preview rail 초안
  - stale session 응답 무효화 로직
- 따라서 2.1 구현은 새로운 truth source를 만들지 말고, Story 1.5에서 만든 manifest/capture readiness/session provider 경계를 확장해야 한다.

### 현재 워크스페이스 상태

- 현재 저장소에는 React/Tauri booth runtime scaffold가 이미 존재한다.
- `src/booth-shell/components/LatestPhotoRail.tsx`와 `src/session-domain/selectors/current-session-previews.ts`가 현재 세션 preview rail의 초기 구현을 담당하고 있다.
- `src/booth-shell/screens/CaptureScreen.test.tsx`에는 현재 세션 사진만 보여야 한다는 기본 검증이 이미 들어 있다.
- 그러나 최신 사진을 명확히 구분하는 표현, review rail 전용 selector 계약, Story 2.2 삭제 흐름이 재사용할 세션 범위 해석은 더 명시적으로 잠글 필요가 있다.

### 이전 스토리 인텔리전스

- Story 1.5는 "저장 truth"와 "preview truth"를 분리하고, 최신 확인용 사진은 preview-ready가 된 뒤에만 rail에 나타나도록 정리했다.
- Story 1.5 구현은 `session.json`과 capture readiness snapshot을 단일 truth 경계로 사용한다. 2.1도 이 구조를 유지해야 하며, 별도 UI 캐시나 전역 이미지 목록을 새로운 진실 계층으로 만들면 안 된다.
- 최근 커밋 `d4a5c15 feat: implement story 1-5 truthful preview waiting flow`와 `09741aa chore: mark story 1-5 done`는 현재 코드 패턴의 직접적인 기준선이다.

### 구현 가드레일

- 활성 세션의 durable truth는 여전히 session-scoped filesystem root와 `session.json`이다. review rail도 이 truth에서 파생돼야 한다. [Source: _bmad-output/planning-artifacts/epics.md#Additional Requirements]
- React UI는 직접 raw host 호출을 늘리지 말고 기존 typed adapter/service와 session provider 경계를 통해 상태를 소비해야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Frontend Architecture]
- booth runtime은 current-session review behavior를 소유하며, 다른 세션 자산이나 mutable authoring state를 고객에게 보여주면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#Domain Requirements]
- cross-session asset leakage는 review, preview, deletion 전체 흐름에서 0이어야 한다. [Source: _bmad-output/planning-artifacts/prd.md#NFR-004 Session Isolation and Privacy]

### 아키텍처 준수사항

- FR-005 구조 매핑의 중심은 `src/booth-shell/`, `src/session-domain/`, `src-tauri/src/session/`이다. [Source: _bmad-output/planning-artifacts/architecture.md#Requirements to Structure Mapping]
- booth review/UI 계산은 프런트엔드 selector 계층에서 고객 안전 copy와 view-model로 변환하고, raw host 상태를 JSX 안에서 직접 풀어내지 않는다. [Source: _bmad-output/planning-artifacts/architecture.md#Pattern Examples]
- session manifest/capture 계약은 `src/shared-contracts/`와 `src-tauri/src/contracts/`가 같은 contract family를 공유해야 한다. review rail용 데이터가 필요해도 별도 중복 DTO를 만들지 않는다. [Source: _bmad-output/planning-artifacts/architecture.md#Project Structure & Boundaries]

### UX 구현 요구사항

- 최신 사진 레일은 현재 세션 썸네일만 가로 스크롤로 보여줘야 하며, 최신 항목이 즉시 식별 가능해야 한다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#UX-DR13]
- 고객 문구는 plain-language, low-density 원칙을 지켜야 하고, filesystem/render-engine/helper 같은 내부 용어를 드러내면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#NFR-001 Customer Guidance Density and Simplicity]
- review rail은 booth-first 고객 흐름의 일부여야 하며, 별도 편집 공간처럼 보이게 만들면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#FR-005 Current-Session Review, Deletion, and In-Session Preset Change]

### 최신 기술 확인 메모

- 현재 로컬 패키지 기준선은 `react 19.2.4`, `react-router-dom 7.13.1`, `zod 4.3.6`, `@tauri-apps/api 2.10.1`이다. 이 스토리는 같은 메이저/마이너 기준선 안에서 구현하고 새 의존성을 추가하지 않는 편이 안전하다.
- React Router 공식 설치 문서는 latest 채널에서 Vite 기반 부트스트랩 뒤 `createBrowserRouter`와 `RouterProvider` 패턴을 계속 안내한다. booth review rail은 라우팅 확장이 아니라 기존 `/booth` surface 내부 상태 확장으로 유지한다. [Source: https://reactrouter.com/start/data/installation]
- Tauri Vite 가이드는 `pnpm dev`, `pnpm build`, `devUrl: http://localhost:5173`, `frontendDist: ../dist`, `src-tauri` watch ignore 구성을 계속 권장한다. review rail 구현은 이 프론트엔드/호스트 경계를 깨지 말아야 한다. [Source: https://v2.tauri.app/start/frontend/vite/]
- Zod 공식 문서는 Zod 4가 stable임을 명시한다. review rail에 필요한 추가 view-model이 있더라도 session/capture의 frozen contract를 우회하는 별도 validation 체계를 만들지 않는다. [Source: https://zod.dev/v4]

### 테스트 요구사항

- 최소 필수 테스트 범위는 아래와 같다.
  - 최신 preview-ready capture가 첫 번째로 정렬되고 시각적으로 구분된다.
  - 다른 세션 capture는 review rail에 나타나지 않는다.
  - preview가 준비되지 않은 capture는 review rail에 나타나지 않는다.
  - stale readiness/capture 응답이 현재 세션 review rail을 오염시키지 않는다.
  - 고객용 rail copy에 internal diagnostic terms가 나타나지 않는다.

### 금지사항 / 안티패턴

- 현재 세션 rail 계산을 전역 이미지 캐시나 다른 세션 목록에서 가져오는 것 금지
- preview-ready가 아닌 capture를 review rail에 미리 노출하는 것 금지
- 최신 항목 강조를 위해 내부 capture id나 filesystem path를 고객 문구에 노출하는 것 금지
- Story 2.2 삭제 정책을 완성하기 전에 삭제 UI를 과도하게 확장하는 것 금지
- React 컴포넌트에서 직접 새로운 `invoke` 호출을 만들며 review data를 우회 조회하는 것 금지
