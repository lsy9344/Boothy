# Story 6.1: 지점별 단계적 배포와 단일 액션 롤백 거버넌스

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

owner / brand operator로서,
선택한 지점 집합에 대해 앱 빌드와 승인된 preset stack을 단계적으로 배포하고 필요 시 한 번에 롤백하고 싶다.
그래서 활성 고객 세션을 끊지 않으면서도 지점 간 일관성과 운영 거버넌스를 유지할 수 있다.

## Acceptance Criteria

1. 새 승인 build 또는 preset stack이 준비된 상태에서 rollout을 시작하면, 시스템은 전체 지점 동시 배포가 아니라 명시적으로 선택한 branch set만 대상으로 삼아야 한다. 또한 branch set, target build, approved preset stack, approval timestamp, actor를 rollout audit에 기록해야 한다.
2. rollout이 승인된 branch set에 적용될 때 각 지점의 승인된 local settings(예: 연락처 정보, bounded operational toggles)는 유지되어야 한다. 또한 이번 전환은 해당 branch set의 build와 preset-stack baseline만 바꿔야 한다.
3. 대상 branch 중 하나라도 active customer session을 가지고 있어서 전환이 세션을 방해할 수 있으면, 시스템은 그 branch에 대한 rollout을 defer 또는 reject해야 한다. 또한 forced update는 적용되지 않아야 하고, 거절/지연 사유는 initiating operator에게 보이며 audit history에도 남아야 한다.
4. rollout 또는 rollback 평가 시 현재 세션이 기존 승인 baseline 위에서 계속 진행되는 branch는 safe transition point 전까지 그 baseline과의 compatibility를 유지해야 한다. 또한 배포 전환이 in-flight session을 무효화하거나 손상시키면 안 된다.
5. promoted branch를 되돌려야 할 때 rollback이 승인되면, 시스템은 last approved build와 approved preset stack을 단일 rollback action으로 복구해야 한다. 또한 이 과정에서도 active customer session을 강제 업데이트로 끊으면 안 된다.
6. rollback 요청 시 approved rollback baseline이 없거나 compatibility check가 실패하면 branch state는 바뀌지 않아야 한다. 또한 initiating operator는 명확한 refusal guidance를 받아야 하고, rejection도 audit에 남아야 한다.
7. rollback이 승인되어 prior baseline이 복구되면, 각 branch는 approved local settings를 유지한 채 last approved build와 preset stack으로 돌아가야 한다. 또한 각 branch가 safe transition point에 도달할 때까지 active-session compatibility 보호가 유지되어야 한다.

## Dev Notes

### 스토리 범위와 목적

- 이 스토리는 Epic 6 / NFR-006의 첫 가치 조각이다. 핵심은 "업데이트 버튼 하나 더 만들기"가 아니라, branch set 단위의 release governance를 host-owned 정책으로 만드는 것이다.
- scope는 build rollout과 approved preset stack rollout을 함께 다루되, active session을 강제로 옮기지 않는 safe transition rule을 제품 기본 약속으로 고정하는 데 있다.
- 이 스토리는 booth customer UX를 바꾸거나 authoring flow를 다시 설계하는 작업이 아니다. privileged rollout/rollback은 internal surface와 host command boundary 안에서만 이뤄져야 한다.

### 스토리 기반 요구사항

- NFR-006은 explicit branch set rollout, one-action rollback, approved local settings preservation, active-session compatibility, 0 forced update를 동시에 요구한다. [Source: _bmad-output/planning-artifacts/prd.md#NFR-006-Safe-Local-Packaging,-Rollout,-and-Version-Pinning]
- Epic 6.1은 rollout audit에 branch set, target build, approved preset stack, approval timestamp, actor를 남기고, active session을 방해하는 branch는 defer/reject 하라고 요구한다. [Source: _bmad-output/planning-artifacts/epics.md#Story-6.1:-지점별-단계적-배포와-단일-액션-롤백-거버넌스]
- release gate는 "branch rollout controls can promote explicitly selected branch sets and roll back the app build and approved preset stack without interrupting an active customer session"을 출시 기준선으로 둔다. [Source: _bmad-output/planning-artifacts/prd.md#Release-Gates]
- branch variance는 approved local settings로 제한되어야 하며, preset catalog, timing rules, core booth journey는 branch 간 일관되어야 한다. [Source: _bmad-output/planning-artifacts/prd.md#NFR-002-Branch-Consistent-Experience]

### 선행 의존성과 구현 순서

- 논리적 선행 순서는 Story 4.4 -> Story 5.3 -> Story 6.1이다.
- 4.4는 preset catalog live pointer와 future-session-only rollback을 이미 정리했다. 6.1의 preset stack rollout/rollback은 이 규칙을 확장해야지 별도 live-version 해석기를 만들면 안 된다.
- 5.3은 host-owned structured audit store와 query seam을 열었다. 6.1의 rollout/rollback audit는 이 저장소/패턴을 재사용하거나 일관된 방식으로 확장하는 편이 안전하다.
- 구현 권장 순서는 다음과 같다.
  - shared contract와 branch baseline shape를 먼저 잠근다.
  - Rust host에 branch-config / rollout repository / compatibility evaluator를 만든다.
  - `/settings` surface에 typed service 기반 UI를 붙인다.
  - 마지막에 release docs/workflow와 runbook을 구현 상태에 맞춰 정리한다.

### 현재 워크스페이스 상태

- `src/settings/screens/SettingsScreen.tsx`는 restricted placeholder이며, "branch-local configuration and runtime flags can be connected here once the admin flow exists." 수준만 갖고 있다. 즉, Epic 6의 가장 자연스러운 entry surface는 있지만 기능은 비어 있다.
- `src/app/routes.tsx`는 이미 `/settings`를 `SurfaceAccessGuard surface=\"settings\"` 뒤에 두고 있다. 따라서 새 privileged rollout UI는 새 top-level route를 만들기보다 이 surface를 확장하는 편이 자연스럽다.
- `src-tauri/src/commands/runtime_commands.rs`와 `src/app/services/capability-service.ts`는 `settings` surface를 admin-auth + runtime-profile 기반으로 허용한다. rollout/rollback도 이 기존 gating을 재사용해야 한다.
- Story 4.4 결과로 `docs/contracts/preset-bundle.md`, `src-tauri/src/preset/preset_catalog_state.rs`, `src/preset-authoring/services/preset-authoring-service.ts`가 live catalog pointer와 rollback command를 이미 갖고 있다.
- Story 5.3 결과로 `src-tauri/src/diagnostics/audit_log.rs`, `src/shared-contracts/schemas/operator-audit.ts`, `src/operator-console/` query seam이 존재한다. rollout audit는 이 기반을 무시하고 별도 비정형 텍스트 로그를 만들면 안 된다.
- 아키텍처는 `src/branch-config/`, `src-tauri/src/branch_config/`, `.github/workflows/release-windows.yml`, `config_store.rs`를 기대하지만 현재 repo에는 해당 경계가 실제로 없다. 6.1 구현은 이 공백을 메우거나, 최소한 문서와 실제 구조를 일치시켜야 한다.
- `docs/release-baseline.md`는 `.github/workflows/release-windows.yml`와 `pnpm build:desktop`, `pnpm release:desktop`를 언급하지만 실제 repo에는 `.github/` 폴더와 해당 스크립트 정의가 없다. 이것은 단순 문서 오타가 아니라 release governance story의 실제 구현 공백이다.
- 현재 git worktree는 많은 modified/untracked 파일을 포함한 dirty state다. 구현 시 기존 변경을 되돌리거나 reset하지 말고, 필요한 경계만 additive 하게 확장해야 한다.

### 관련 이전 스토리 인텔리전스

- Story 4.4는 "latest published version이 곧 live catalog version"이라는 가정을 폐기하고 host-owned catalog state pointer를 도입했다. 6.1의 preset stack rollout도 같은 원칙으로 live deployment baseline을 pointer/state로 관리해야 한다. [Source: _bmad-output/implementation-artifacts/4-4-미래-세션-대상-롤백과-카탈로그-버전-관리.md#구현-가드레일]
- Story 4.4는 active session의 `manifest.activePreset`과 `catalogSnapshot`을 rollback side effect로 바꾸지 않도록 잠갔다. branch rollout도 동일하게 active session manifest를 다시 쓰면 안 된다. [Source: docs/contracts/session-manifest.md#필드-규칙]
- Story 5.3은 lifecycle/intervention/publication/rollback 이벤트를 host-owned structured audit store로 수렴시켰다. rollout/rollback 거버넌스도 console log나 산발적 JSON 대신 이 감사 패턴을 재사용해야 한다. [Source: _bmad-output/implementation-artifacts/5-3-라이프사이클-개입-복구-감사-로그-기록.md#구현-가드레일]
- Story 5.3은 audit write failure가 session/preset truth를 깨지 않도록 atomic guard를 요구했다. rollout audit도 branch baseline mutation과 audit persistence를 분리해 같은 보호를 가져가야 한다. [Source: _bmad-output/implementation-artifacts/5-3-라이프사이클-개입-복구-감사-로그-기록.md#테스트-요구사항]

### Git / 워크스페이스 인텔리전스

- 최근 `git log -5 --oneline`는 story 1 계열 커밋과 `chore: commit all pending changes`만 보여 주므로, Epic 4~6 관련 최신 패턴은 커밋 히스토리보다 현재 워크스페이스 파일에서 읽는 편이 정확하다.
- 실제 rollout/preset/audit 관련 구현은 현재 uncommitted 변경에도 많이 포함되어 있으므로, dev agent는 오래된 commit title보다 현재 파일 시스템 상태를 진실의 기준으로 삼아야 한다.

### 구현 가드레일

- rollout/rollback 판단은 React local state나 CI metadata만으로 확정하면 안 된다. active session 여부, compatibility, local settings preservation 여부는 Rust host가 authoritative 하게 판정해야 한다.
- build rollout과 preset stack rollout을 분리 구현하더라도 branch-visible baseline은 하나의 approved deployment intent로 묶여야 한다. build만 바꾸고 preset stack 감사가 누락되거나, 반대로 preset stack만 바뀌는데 branch baseline 기록이 남지 않는 상태를 만들면 안 된다.
- active customer session이 있는 branch에 대해 updater auto-apply, forced restart, live preset rebinding 같은 식의 강제 전환을 만들면 안 된다.
- approved local settings는 rollout payload merge 과정에서 덮어쓰거나 reset하면 안 된다. 연락처 정보와 bounded operational toggles는 branch-local config에 남고, rollout이 바꾸는 것은 approved build/preset baseline뿐이어야 한다.
- booth customer surface와 일반 booth route에는 rollout status, branch list, release jargon, internal refusal reason을 노출하면 안 된다.
- operator/audit/history를 위해 raw filesystem path, full helper output, CI secret 값, certificate path 같은 민감 정보를 UI 기본 payload에 담으면 안 된다.
- release-governance 구현을 핑계로 Story 4.4의 immutable preset bundle contract를 깨거나, Story 5.3의 audit taxonomy와 충돌하는 별도 로그 체계를 만들면 안 된다.

UI surface는 `/settings`를 재사용하되, 도메인 코드는 settings 화면에 직접 뒤섞지 않는 편이 아키텍처에 맞다. [Source: _bmad-output/planning-artifacts/architecture.md#Requirements-to-Structure-Mapping]

### UX 구현 요구사항

- rollout/rollback surface는 internal density를 가질 수 있지만, branch 선택, target baseline, local settings preservation, active-session defer/reject 여부가 한눈에 읽혀야 한다.
- confirmation copy는 "지금 진행 중인 세션은 유지되고, 안전한 전환 시점 이후에만 적용된다"는 제품 약속을 분명히 보여 줘야 한다.
- refusal/defer copy는 operator가 다시 시도할 수 있는 조건을 이해하게 해야 하지만, raw error text나 내부 enum 이름을 그대로 노출하면 안 된다.
- customer booth surface는 기존 copy budget과 internal-term 금지 규칙을 계속 지켜야 한다. rollout governance 정보는 booth UI에 섞이면 안 된다.

### 최신 기술 확인 메모

- 현재 로컬 기준선은 `react 19.2.4`, `react-router-dom 7.13.1`, `@tauri-apps/api 2.10.1`, `zod 4.3.6`, `tauri 2.10.3`이다. 이번 스토리는 새 프레임워크 도입보다 기존 typed boundary와 Tauri host 경계를 확장하는 편이 안전하다. [Source: package.json] [Source: src-tauri/Cargo.toml]
- Tauri v2 공식 문서는 privileged backend work를 `#[tauri::command]` 경계로 두는 패턴을 전제로 한다. rollout/rollback은 low-frequency privileged mutation이므로 command 중심 구현이 적합하다. [Source: https://v2.tauri.app/develop/calling-rust/]
- Tauri capability 문서는 capability file을 통한 window/surface 권한 분리를 설명한다. rollout governance는 기존 `settings` capability/profile gate를 재사용하고 booth surface로 우회 노출되면 안 된다. [Source: https://v2.tauri.app/reference/acl/capability/]
- Tauri updater 공식 문서는 updater plugin이 앱 내부 update flow를 제공한다고 설명한다. Boothy의 "0 forced update during active session" 요구사항을 고려하면, auto-apply updater를 도입하기보다 safe transition point를 host policy로 강제하는 편이 맞다. 이것은 공식 문서와 PRD를 결합한 구현 추론이다. [Source: https://v2.tauri.app/plugin/updater/] [Source: _bmad-output/planning-artifacts/prd.md#NFR-006-Safe-Local-Packaging,-Rollout,-and-Version-Pinning]
- `tauri-action` 공식 저장소 README는 GitHub Action이 앱을 native binary로 build 하고 release 업로드까지 담당할 수 있음을 보여 준다. repo에 workflow가 아직 없다면, story 범위 안에서 최소 draft release workflow를 추가하거나 docs를 실제 범위로 축소해야 한다. [Source: https://github.com/tauri-apps/tauri-action]
- Zod 4 stable baseline은 기존 shared schema 전략과 잘 맞는다. rollout input/result, compatibility verdict, rejection reason도 ad-hoc JSON parsing 대신 shared schema family로 두는 편이 안전하다. [Source: https://zod.dev/]

### 테스트 요구사항

- 최소 필수 테스트 범위:
  - rollout이 explicit branch set만 대상으로 삼고 audit에 branch set / build / preset stack / actor / timestamp를 남기는지
  - active session이 있는 branch는 defer/reject 되고 forced update가 일어나지 않는지
  - safe transition point 이후에만 branch baseline이 실제로 전환되는지
  - rollback이 last approved build + preset stack을 함께 복원하고 approved local settings는 유지하는지
  - approved rollback baseline이 없거나 compatibility check가 실패하면 mutation 없이 거절되는지
  - preset stack rollout 후 active session의 `catalogSnapshot`과 preset binding이 유지되고 새 session만 새 baseline을 보는지
  - `/settings` capability gate 밖에서는 rollout UI가 노출되지 않는지
  - audit persistence failure 또는 malformed release baseline이 기존 booth/operator/authoring 흐름을 깨지 않는지

### 금지사항 / 안티패턴

- active session 중 updater auto-install, app restart 강제, live preset rebinding으로 rollout을 해결하려는 것 금지
- branch-local settings를 release payload apply 과정에서 덮어쓰거나 삭제하는 것 금지
- rollout governance를 `operator-console` 또는 `preset-authoring` 내부에 임시로 끼워 넣고 `branch-config` 경계를 영구히 생략하는 것 금지
- component에서 직접 `invoke`를 호출하고 shared contract parsing을 건너뛰는 것 금지
- session manifest나 preset bundle을 deployment truth 저장소로 승격시키는 것 금지
- `.github/workflows/release-windows.yml` 부재를 무시한 채 문서만 "있다"고 유지하는 것 금지
