# Story 2.4: 조정된 종료 시각 표시와 경고/종료 알림

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

booth customer로서,
내 세션에 남은 시간이 얼마나 되는지와 종료에 가까워졌을 때 어떤 일이 일어나는지 정확히 알고 싶다.
그래서 추측하지 않고 자신 있게 촬영을 마무리할 수 있다.

## Acceptance Criteria

1. 활성 세션이 `Session Timing Policy`를 따를 때 고객이 세션 시작 후 부스 흐름에 들어오면, 조정된 종료 시각이 세션 시작부터 보여야 한다. 또한 시간 표시는 고객 안전 문구와 부스 UI에 맞는 시각 위계를 사용해야 한다.
2. 세션이 조정된 종료 시각 5분 전 구간에 들어가면, 부스는 승인된 경고음을 재생하고 눈에 띄는 경고 상태를 보여야 한다. 또한 고객은 이 시점에도 촬영이 계속 가능한지 이해할 수 있어야 한다.
3. 세션이 조정된 종료 시각에 도달하면, 부스는 승인된 종료 알림음을 재생하고 촬영 종료 여부를 반영한 안내로 갱신되어야 한다. 또한 상태 전환 과정에서 scheduler, policy 같은 내부 용어를 노출하면 안 된다.
4. 경고 또는 조정된 종료 시각 임계값에 도달한 qualifying session을 운영 환경 기준으로 평가할 때, 5분 경고와 정확 종료 알림은 99% 세션에서 +/- 5초 이내에 발생해야 한다. 또한 승인된 연장 적용과 로그 기록이 없는 한 종료 후 촬영 시도는 차단되어야 한다.

## Dev Notes

### 스토리 범위와 목적

- 이 스토리는 Epic 2 / FR-006의 가치 조각으로, 고객이 세션 시작부터 종료까지 시간을 믿고 행동할 수 있게 만드는 작업이다.
- 핵심은 "타이머를 예쁘게 보여주는 것"이 아니라, host가 계산한 종료 진실을 고객 화면, 경고음, 촬영 가능 여부에 일관되게 반영하는 것이다.
- Story 3.1이 종료 직후 `Export Waiting` / `Completed` / `Phone Required` 중 하나로 명시적으로 진입시키는 작업이라면, Story 2.4는 그 직전의 warning/exact-end 경험과 종료 차단 경계를 먼저 안정화하는 단계다.

### 스토리 기반 요구사항

- `Session Timing Policy` minimum baseline:
  - 조정된 종료 시각은 승인된 timing input으로 세션 시작 시 고정되고 세션 내내 보여야 한다.
  - 경고는 T-5분, 정확 종료 동작은 T=0 기준으로 일어나야 한다.
  - 승인된 operator extension이 적용되고 로그에 남지 않은 한 종료 후 새 촬영은 차단되어야 한다.
- 고객은 경고와 종료 시점 모두에서 "지금 촬영 가능한지", "이제 기다려야 하는지/마무리해야 하는지"를 plain-language로 이해할 수 있어야 한다.
- 사운드 알림은 브랜드 경험을 돕는 보조 장치이지, 고객을 놀라게 하거나 공포를 키우는 신호가 되면 안 된다.

### 선행 의존성과 구현 순서

- 선행 구현은 Story 1.2 ~ 1.5, Story 2.1, Story 2.2다.
- 특히 현재 기준선에는 다음이 이미 있다.
  - `session.json` 기반 durable session truth
  - `CaptureScreen` + `SessionProvider` + typed capture adapter 경계
  - current-session review rail / deletion / stale-session guard 패턴
  - `capture-readiness` 계약 안의 `warning`, `export-waiting`, `completed`, `phone-required` reason code
- 반면 아키텍처가 기대하는 `src/timing-policy/`, `src-tauri/src/timing/`, `timing_commands.rs` 같은 전용 timing 경계는 아직 없다. 이번 스토리는 그 간극을 메우는 첫 구현이 될 가능성이 높다.
- Story 2.3은 이미 `ready-for-dev` 상태지만 아직 구현되지 않았으므로, 2.4 구현은 "프리셋 변경 UI가 완성돼 있다"는 가정 없이 현재 booth flow 안에 timing 경험을 끼워 넣어야 한다.

### 현재 워크스페이스 상태

- 현재 프런트엔드의 booth runtime은 `CaptureScreen`이 `ReadinessScreen`, `PreviewWaitingPanel`, `LatestPhotoRail`을 조합해 상태를 보여주는 구조다.
- `src/capture-adapter/services/capture-runtime.ts`는 `get_capture_readiness`, `request_capture`, `delete_capture`를 typed gateway로 감싸고, readiness는 `listen('capture-readiness-update')`와 1.5초 polling으로 유지한다.
- Rust host의 `src-tauri/src/capture/normalized_state.rs`는 lifecycle stage가 `warning`, `export-waiting`, `completed`, `phone-required`일 때 customer-safe readiness DTO를 만들 수 있다.
- 그러나 현재 manifest/DTO에는 session-level adjusted end time 자체를 전달하는 canonical field가 없고, lifecycle을 실제로 `warning`으로 전환시키는 timing policy module도 없다.

### 이전 스토리 인텔리전스

- Story 2.1과 2.2는 현재 세션 범위, stale/foreign update 차단, manifest-first truth, customer-safe wording이라는 네 가지 경계를 강하게 잠갔다.
- Story 2.2 구현은 capture/delete mutation도 `CAPTURE_PIPELINE_LOCK` 아래에서 직렬화하고, asset staging 후 manifest commit 순서를 지켜 partial failure 회귀를 막았다.
- 따라서 2.4도 UI 로컬 타이머나 ad-hoc timeout만으로 진실을 만들면 안 되며, timing 전환이 실제 capture readiness와 충돌할 때는 host-normalized truth가 우선해야 한다.
- 사운드/경고 상태도 rerender나 재구독 때문에 여러 번 발화되면 customer trust를 해친다. timing 이벤트는 same-session, threshold-crossing, one-shot 성격을 기준으로 다뤄야 한다.

### 구현 가드레일

- 조정된 종료 시각과 종료 후 캡처 차단은 host 소유 진실이어야 한다. React는 표시와 고객 문구를 맡되 종료 판정을 단독으로 소유하면 안 된다. [Source: _bmad-output/planning-artifacts/epics.md#Additional Requirements]
- booth runtime은 timing guidance를 제품 약속으로 소유하며, branch-local 임의 규칙이나 generic slot rule을 발명하면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#FR-006 Coupon-Adjusted Timing, Warning Alerts, and Exact-End Behavior]
- 현재 세션 review/delete 흐름과 마찬가지로, timing update도 foreign session payload가 현재 세션을 덮지 못하게 세션 일치성 검증을 유지해야 한다.
- 종료 시점 안내는 "촬영이 끝났어요", "잠시만 기다려 주세요", "도움이 필요해요"처럼 고객 행동 중심이어야 하며 scheduler, timeout, policy 같은 내부 용어를 노출하면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#NFR-001 Customer Guidance Density and Simplicity]

### 아키텍처 준수사항

- FR-006의 기본 구조 매핑은 `src/timing-policy/` 와 `src-tauri/src/timing/` 이다. 실제 구현이 임시로 다른 경계를 건드리더라도 이 책임 분리를 향해야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Requirements to Structure Mapping]
- frontend-to-host timing 흐름은 Tauri command / event 또는 channel 경계를 통해 typed adapter로 들어와야 하며, JSX에서 raw `invoke`나 window timer를 직접 진실처럼 사용하면 안 된다. [Source: _bmad-output/planning-artifacts/architecture.md#API & Communication Patterns]
- customer-facing state projection은 같은 normalized host truth에서 파생돼야 하며, booth copy와 operator/audit truth가 서로 다른 ad-hoc 계산을 쓰면 안 된다. [Source: _bmad-output/planning-artifacts/architecture.md#State Management Patterns]
- SQLite는 timing transition과 lifecycle audit 저장소일 수 있지만 photo/session durable truth 자체를 대신하면 안 된다. [Source: _bmad-output/planning-artifacts/architecture.md#Data Architecture]

### UX 구현 요구사항

- 조정된 종료 시각은 세션 시작부터 명확히 보여야 하고, 5분 전 경고와 종료 후 다음 행동은 고객 안전 문구로 이해 가능해야 한다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#구속되는-ux-요구사항]
- 시간 안내 컴포넌트는 디지털 타이머와 상태별 시각 강조를 제공해야 하며, 5분 전과 종료 시점에 사운드 알림과 함께 동작해야 한다. [Source: _bmad-output/planning-artifacts/epics.md#UX-DR12]
- warning 상태는 촬영 흐름을 과도하게 깨지 않는 보조 정보여야 하며, exact-end 이후에는 다음 행동 위계가 분명해야 한다. [Source: _bmad-output/planning-artifacts/ux-design-specification.md#Journey-Patterns]
- 고객 화면의 카피 밀도는 낮게 유지하고, 동적 시간 값 외에는 핵심 지시 1개, 보조 문장 1개, 주요 액션 1개 원칙을 넘기지 않는 것이 안전하다. [Source: _bmad-output/planning-artifacts/prd.md#NFR-001 Customer Guidance Density and Simplicity]

### 최신 기술 확인 메모

- 현재 로컬 패키지 기준선은 `react 19.2.4`, `react-router-dom 7.13.1`, `zod 4.3.6`, `@tauri-apps/api 2.10.1`이다. 이번 스토리는 새 의존성 추가보다 이 기준선 안에서 timing domain을 세우는 편이 안전하다.
- React 공식 문서는 `useEffectEvent`를 통해 timer/event-listener 안에서 최신 state를 읽되 effect 재구독을 피하는 패턴을 안내한다. timing countdown, warning sound trigger, event listener cleanup에 적합하다. [Source: https://react.dev/reference/react/useEffectEvent]
- Tauri 공식 문서는 SPA에서 `listen`으로 등록한 이벤트 리스너는 scope가 끝날 때 직접 `unlisten` 해야 한다고 안내한다. timing listener는 component unmount나 session 교체 시 해제돼야 중복 경고음이 쌓이지 않는다. [Source: https://v2.tauri.app/develop/_sections/frontend-listen/]
- Tauri 공식 문서는 이벤트 시스템보다 channels가 ordered data stream에 더 적합하다고 설명한다. 초당 tick을 host에서 계속 밀어야 한다면 channel 또는 host truth + local countdown 보간 중 하나를 의식적으로 선택해야 한다. 이것은 문서와 현재 아키텍처를 바탕으로 한 구현 추론이다. [Source: https://v2.tauri.app/develop/calling-frontend/]
- Tauri 공식 문서는 `mockIPC(..., { shouldMockEvents: true })` 로 Rust emit 이벤트를 Vitest에서 시뮬레이션할 수 있다고 안내한다. warning/end timing 이벤트 회귀 테스트에 그대로 활용할 수 있다. [Source: https://v2.tauri.app/develop/tests/mocking/]
- Zod 공식 문서는 Zod 4가 stable이라고 명시한다. timing DTO도 기존 shared schema family 안에서 검증을 추가하고 ad-hoc runtime parsing을 새로 만들지 않는 편이 안전하다. [Source: https://zod.dev/]

### 테스트 요구사항

- 최소 필수 테스트 범위:
  - 세션 시작 직후 adjusted end time이 보인다.
  - T-5에서 warning visual state와 sound trigger가 한 번만 발생한다.
  - T=0 이후 customer guidance가 "촬영 종료" 의미로 바뀌고 `requestCapture`가 차단된다.
  - foreign timing update 또는 stale threshold event가 현재 세션 상태를 덮지 못한다.
  - 종료 후에는 post-end state 전 스텝이라도 ambiguous ready state로 되돌아가지 않는다.
  - 고객 문구에 coupon formula, scheduler, policy/internal error 용어가 나타나지 않는다.

### 금지사항 / 안티패턴

- 조정된 종료 시각을 프런트 로컬 시계만으로 독자 계산해 host truth와 분기시키는 것 금지
- rerender/route re-entry마다 warning sound 또는 end alert를 반복 재생하는 것 금지
- 종료 후 차단을 UI disabled 상태만으로 구현하고 host가 실제 capture를 허용하는 상태로 남겨두는 것 금지
- Story 3.1 범위인 `Export Waiting` / `Completed` / `Phone Required` 완성 UX를 이번 스토리 안에서 과도하게 선구현하는 것 금지
- Story 2.3이 아직 없다는 이유로 timing UI 안에 프리셋 변경 로직을 끼워 넣거나 기존 preset visibility를 깨뜨리는 것 금지
