## Story

owner / brand operator로서,
구현 전에 공통 계약을 먼저 동결하고 싶다,
그래서 booth, operator, authoring, host 구현이 같은 기준을 따를 수 있다.

## Acceptance Criteria

1. `session.json` session manifest schema가 explicit version을 포함한 기준선으로 동결되어야 하고, capture correlation, preset version reference, post-end state 필드 기준이 문서와 shared schema에서 일치해야 한다.
2. preset bundle schema와 runtime profile / capability model이 함께 확정되어야 하며, booth, operator, authoring surface가 같은 계약을 참조해야 한다.
3. error envelope와 helper/sidecar protocol contract가 확인 가능해야 하고, 개발팀이 바로 검증할 수 있는 테스트 가능한 예시 또는 실행 가능한 검증 기준이 함께 남아야 한다.

## Dev Notes

### 스토리 범위와 제품 목적

- 이 스토리는 고객 기능을 직접 늘리는 작업이 아니라, 이후 기능 구현이 같은 제품 기준으로 움직이게 하는 foundational contract freeze다.
- 목표는 새 계약을 과하게 발명하는 것이 아니라, 이미 등장한 session / preset / helper / capability / error semantics를 authoritative baseline으로 닫는 것이다.
- Story 1.15는 Canon helper profile과 publication contract의 deeper implementation detail을 소유하고, Story 1.16은 build/release baseline을 소유한다. Story 1.14는 그 선행 공통 기준만 잠근다.

### 제품/아키텍처 가드레일

- session truth는 계속 `session.json`과 session-scoped filesystem root가 소유해야 한다. React나 SQLite가 durable truth를 대신 소유하면 안 된다. [Source: _bmad-output/planning-artifacts/architecture.md#Data Architecture] [Source: docs/contracts/session-manifest.md]
- preset truth는 immutable published bundle과 capture-bound preset version이 소유해야 한다. live catalog pointer나 display name만으로 런타임 해석을 대신하면 안 된다. [Source: _bmad-output/planning-artifacts/architecture.md#Data Architecture] [Source: docs/contracts/preset-bundle.md]
- helper truth는 host-normalized projection 뒤에 있어야 한다. booth/operator가 raw helper vocabulary를 직접 소비하면 안 된다. [Source: _bmad-output/planning-artifacts/architecture.md#API & Communication Patterns] [Source: docs/contracts/camera-helper-sidecar-protocol.md]
- host boundary 응답은 typed payload 또는 typed error envelope를 사용해야 하고, ad hoc object return을 늘리면 안 된다. [Source: _bmad-output/planning-artifacts/architecture.md#Format Patterns]
- capability gating은 booth 기본 surface를 유지하되, operator/authoring/settings 노출을 runtime profile과 host authority 기준으로 제한해야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Authentication & Security]

### 구현 가드레일

- Story 1.14에서 Canon EDSDK runtime detail 전체를 다시 설계하지 말 것. 그 deeper helper profile은 Story 1.15의 책임이다.
- authoring publication payload/state machine 세부를 여기서 과도하게 확장하지 말 것. 1.14는 shared baseline과 reference point를 잠그고, 1.15/4.x가 deeper publication behavior를 닫게 둔다.
- 새 schemaVersion을 가볍게 올리지 말 것. 현재 baseline은 freeze story이므로, 가능한 한 `v1` authoritative shape를 명확히 하고 drift를 제거하는 데 집중한다.
- 문서만 고치고 실행 검증을 남기지 말 것. 이 스토리의 핵심 가치는 "문서 + schema + test"가 함께 잠기는 것이다.
- customer-facing copy나 UX state 이름을 기술 편의로 바꾸지 말 것. contract freeze는 내부 경계 명확화이지 고객 어휘 변경이 아니다.
