# Story 5.3: 라이프사이클, 개입, 복구 감사 로그 기록

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

owner / operations lead로서,
라이프사이클과 운영자 개입 이벤트가 일관되게 기록되길 원한다.
그래서 장애, 복구, 종료 결과, 지원 부담을 지점별로 감사하고 회고할 수 있다.

## Acceptance Criteria

1. host가 세션 lifecycle transition, operator intervention, publication-related recovery event, critical failure를 최종 확정하면 시스템은 timestamp, actor 또는 source, session reference, event type을 포함한 audit record를 남겨야 한다. 또한 이 기록은 operator가 support와 retrospective review에 사용할 수 있게 queryable해야 한다.
2. operator나 owner가 session history를 검토할 때 state transition, intervention attempt, final outcome을 구분할 수 있어야 한다. 또한 audit log는 durable photo/session asset truth와 분리된 store에 유지되어야 한다.

## Dev Notes

### 스토리 범위와 목적

- 이 스토리는 Epic 5 / FR-009의 세 번째 가치 조각이다. 핵심은 새 operator action을 더 만드는 것이 아니라, 이미 host가 알고 있는 lifecycle truth와 recovery truth를 감사 가능한 형태로 모으는 것이다.
- 목표는 "세션 폴더에 로그를 더 쌓는 것"이 아니라, 지점 운영과 retrospective review에 쓸 수 있는 host-owned structured audit view를 만드는 것이다.
- 이 스토리는 customer booth flow를 다시 설계하거나, session/photo asset truth를 SQLite로 이전하는 작업이 아니다.

### 스토리 기반 요구사항

- FR-009은 operator가 blocked state를 진단하고 bounded recovery를 수행할 수 있도록 diagnostics, actions, lifecycle visibility를 제공해야 한다고 정의한다. [Source: _bmad-output/planning-artifacts/prd.md#FR-009-Operational-Safety-and-Recovery]
- Epic 5.3은 lifecycle transition, operator intervention, publication-related recovery event, critical failure를 timestamp, actor/source, session reference, event type과 함께 기록하고 queryable하게 유지하라고 요구한다. [Source: _bmad-output/planning-artifacts/epics.md#Story-5.3:-라이프사이클,-개입,-복구-감사-로그-기록]
- architecture baseline은 SQLite가 lifecycle, timing transition, operator intervention, preset publication audit, rollout history를 저장하되 photo/session artifact truth를 소유하면 안 된다고 못 박는다. [Source: _bmad-output/planning-artifacts/architecture.md#Data-Architecture]
- customer-facing surfaces는 raw diagnostics와 internal recovery steps를 노출하면 안 되고, operator-safe detail만 별도 projection으로 보여야 한다. [Source: _bmad-output/planning-artifacts/prd.md#Named-Policy-References] [Source: _bmad-output/planning-artifacts/architecture.md#API-&-Communication-Patterns]
- 로그와 diagnostics도 최소한의 session-identifying data만 남겨야 하고 cross-session privacy를 해치면 안 된다. [Source: _bmad-output/planning-artifacts/prd.md#NFR-004-Session-Isolation-and-Privacy] [Source: _bmad-output/planning-artifacts/architecture.md#Authentication-&-Security]

### 선행 의존성과 구현 순서

- 논리적 선행 순서는 Story 5.1 -> 5.2 -> 5.3이다.
- 5.1은 operator가 current session context와 blocked-state diagnostics를 보게 하고, 5.2는 approved recovery action을 실행하게 한다.
- 현재 epic 5 story file은 5.1과 5.2가 아직 생성되지 않았으므로, 5.3 구현 시 operator action UI가 비어 있더라도 host event write seam과 query seam은 먼저 구축할 수 있다.
- 권장 구현 순서는 다음과 같다.
  - shared audit contract와 repository shape를 먼저 잠근다.
  - Rust host에 structured audit store와 write/query 경계를 만든다.
  - 기존 timing/handoff/publication/rollback 경계에서 audit emit을 연결한다.
  - 마지막에 `/operator` query UI를 붙인다.

### 현재 워크스페이스 상태

- `/operator` route와 capability gate는 이미 존재하지만, `src/operator-console/screens/OperatorSummaryScreen.tsx`는 아직 placeholder다.
- 현재 Rust host에는 `src-tauri/src/commands/operator_commands.rs`나 `src-tauri/src/diagnostics/` 모듈이 없다. `src-tauri/src/lib.rs`의 invoke handler도 operator-specific command를 등록하지 않는다.
- session root는 이미 `diagnostics/` 디렉터리를 만들고, `src-tauri/src/timing/mod.rs`는 `warning`, `ended`, `extension-hook-reserved`를 `diagnostics/timing-events.log`에 append한다.
- `src-tauri/src/handoff/mod.rs`도 post-end evaluation을 같은 `diagnostics/timing-events.log`에 append한다. 즉, lifecycle 로그가 현재는 per-session text file에 분산돼 있다.
- preset publication/rollback 쪽은 이미 host-owned audit artifact를 부분적으로 가지고 있다.
  - `src-tauri/src/preset/authoring_pipeline.rs`는 `preset-authoring/publication-audit/{presetId}.json`에 publish/reject 기록을 남긴다.
  - `src-tauri/src/preset/preset_catalog_state.rs`는 `preset-catalog/catalog-audit/{presetId}.json`과 catalog revision history를 관리한다.
- `src-tauri/Cargo.toml`에는 아직 SQLite 전용 crate가 선언되어 있지 않고, 현재 host는 JSON/text 기반 persistence 위주다. architecture 기대와 실제 구현 사이의 이 간극을 5.3이 메우는 것이 자연스럽다.
- `src/shared-contracts/events/index.ts`에는 `capture-readiness-update`와 surface request 이벤트만 있고, audit history query/result DTO나 operator history event family는 아직 없다.

### 관련 이전 스토리 인텔리전스

- Story 2.4는 warning/end timing truth를 host가 계산하고 `timing-events.log`로 남기는 패턴을 이미 만들었다. 5.3은 이 이벤트 이름과 상태 의미를 재사용하고, query surface만 structured하게 승격하는 편이 안전하다. [Source: _bmad-output/implementation-artifacts/2-4-조정된-종료-시각-표시와-경고-종료-알림.md#구현-가드레일]
- Story 3.3은 `phone-required`와 `completed/handoff-ready`를 host-normalized post-end truth로 잠갔다. audit log도 raw failure dump 대신 이 normalized outcome을 기록해야 operator retrospective가 일관된다. [Source: _bmad-output/implementation-artifacts/3-3-handoff-ready와-phone-required-보호-안내.md#구현-가드레일]
- Story 4.3은 publication success/rejection audit를 host-owned JSON으로 남기고 active session non-mutation을 지켰다. 5.3은 이 publication audit semantics를 버리지 말고 중앙 audit query seam에 연결해야 한다. [Source: _bmad-output/implementation-artifacts/4-3-승인과-불변-게시-아티팩트-생성.md#구현-가드레일]
- Story 4.4는 rollback과 catalog audit를 future-session-safe pointer update로 정리했다. 5.3의 publication-related recovery audit도 active session preset binding을 건드리지 않는 같은 원칙을 따라야 한다. [Source: _bmad-output/implementation-artifacts/4-4-미래-세션-대상-롤백과-카탈로그-버전-관리.md#구현-가드레일]

### 구현 가드레일

- structured audit store는 session folder, `session.json`, photo asset, preset bundle, catalog state의 primary truth를 대체하면 안 된다.
- operator queryability를 위해 기존 scattered text/JSON artifact를 읽어 합치는 임시 어댑터만 만들고 끝내면 안 된다. 새 central query seam이 authoritative operator history source가 되어야 한다.
- customer-facing booth surface는 이번 스토리로 인해 raw diagnostics, operator action detail, internal recovery taxonomy를 보게 되면 안 된다.
- audit payload에는 꼭 필요한 correlation data만 넣고, raw filesystem path, full helper stderr, cross-session identifier dump를 기본 필드로 저장하면 안 된다.
- 기존 publication audit와 catalog audit를 없애거나 bundle immutability를 깨는 방향으로 "통합"하면 안 된다. 필요하면 mirror/write-through 하되 의미를 일치시켜야 한다.
- operator action history를 session manifest 안에 끼워 넣거나 React local state만으로 유지하면 안 된다. privileged host command + host-owned store가 기준이어야 한다.

### 아키텍처 준수사항

- FR-009 기본 구조 매핑은 `src/operator-console/`, `src/diagnostics-log/`, `src-tauri/src/diagnostics/`다. 현재 repo에 일부 폴더가 비어 있더라도 이 경계를 우선 따라야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Requirements-to-Structure-Mapping]
- React component는 typed adapter/service를 통해서만 host query command를 호출해야 하고, component에서 직접 `invoke`를 추가하면 안 된다. [Source: _bmad-output/planning-artifacts/architecture.md#Boundary-rule]
- host authority 규칙상 privileged operator query/action, session mutation, publication/rollback-sensitive write는 Rust host가 소유해야 한다. [Source: _bmad-output/planning-artifacts/architecture.md#Authentication-&-Security]
- frontend-to-host는 Tauri command, host-to-frontend ordered state는 channel/event 경계를 따라야 한다. 이 스토리의 기본 query는 command 중심으로 시작하는 편이 자연스럽다. [Source: _bmad-output/planning-artifacts/architecture.md#API-&-Communication-Patterns]
- customer-facing state와 operator-facing diagnosis는 같은 normalized host truth에서 파생돼야 하며, 별도 ad-hoc 분기나 raw helper interpretation을 만들면 안 된다. [Source: _bmad-output/planning-artifacts/architecture.md#Core-Architectural-Decisions]

### UX 구현 요구사항

- operator history는 customer-safe copy와 다른 밀도를 가질 수 있지만, 여전히 raw helper dump 대신 category, next action, outcome 중심으로 보여야 한다.
- history panel은 state transition, intervention attempt, final outcome이 한눈에 구분되도록 시각적 grouping이 필요하다.
- `Phone Required`나 post-end failure 관련 정보가 operator surface에 보이더라도, customer가 봐야 할 booth copy와 혼동되지 않게 분리해야 한다.
- 고객 화면에는 새 audit affordance, diagnostics jargon, operator history 요약이 절대 노출되면 안 된다.

### 최신 기술 확인 메모

- 현재 로컬 기준선은 `react 19.2.4`, `react-router-dom 7.13.1`, `@tauri-apps/api 2.10.1`, `zod 4.3.6`, `tauri 2.10.3`이다. 이번 스토리는 새 프레임워크 도입보다 기존 typed boundary와 Tauri host 경계를 확장하는 편이 안전하다. [Source: package.json] [Source: src-tauri/Cargo.toml]
- Tauri v2 공식 문서는 privileged backend work를 `#[tauri::command]`로 노출하고 invoke handler에 등록하는 패턴을 전제로 한다. operator audit query와 intervention log 조회도 low-frequency privileged query이므로 command 경계가 맞다. [Source: https://v2.tauri.app/develop/calling-rust/]
- Tauri capability 문서는 window별 capability file과 permission 구성을 사용한다. operator history는 기존 `/operator` surface와 capability gate를 재사용해야 하며 booth surface로 우회 노출되면 안 된다. [Source: https://v2.tauri.app/reference/acl/capability/]
- Tauri frontend 문서는 channels가 높은 빈도의 ordered stream에 더 적합하다고 설명한다. 이번 스토리는 history query가 중심이므로 command-first가 적절하고, 실시간 tail이 필요해질 때만 별도 stream을 검토하는 편이 안전하다. 이것은 공식 문서와 현재 요구사항을 바탕으로 한 구현 추론이다. [Source: https://v2.tauri.app/develop/calling-frontend/]
- React 공식 문서는 `useEffectEvent`가 effect와 event를 분리하기 위한 Hook이라고 설명한다. operator console에서 polling/subscription side effect를 정리할 때는 유용하지만, service identity나 dependency shortcut 대체재로 오용하면 안 된다. [Source: https://react.dev/reference/react/useEffectEvent]
- Zod 공식 사이트는 Zod 4가 stable이라고 명시한다. audit query/result도 ad-hoc parsing 대신 shared schema family로 추가하는 편이 안전하다. [Source: https://zod.dev/packages/zod]

### 테스트 요구사항

- 최소 필수 테스트 범위:
  - warning/end/post-end/phone-required 같은 lifecycle outcome이 structured audit store에 남는지
  - publication success/rejection, catalog rollback이 기존 JSON audit 외에도 central audit seam에서 조회되는지
  - operator query가 session filter, event type filter, 시간 역순 정렬을 안정적으로 제공하는지
  - audit write 실패가 session manifest, capture asset, published bundle, catalog state를 깨지 않는지
  - `/operator` route 밖에서는 history query UI가 노출되지 않는지
  - foreign session response나 stale query 결과가 현재 operator 화면을 오염시키지 않는지
  - customer booth flow는 이번 스토리 이후에도 기존 capture/delete/post-end behavior를 그대로 유지하는지

### 금지사항 / 안티패턴

- `diagnostics/timing-events.log`만을 operator history의 최종 query source로 삼는 것 금지
- session manifest에 audit array를 추가해 durable session truth와 운영 로그를 섞는 것 금지
- published bundle 또는 catalog state를 수정/삭제하면서 audit 통합을 해결하려는 것 금지
- operator component에서 직접 `invoke`를 호출하고 shared-contracts parsing을 건너뛰는 것 금지
- raw helper output, absolute asset path, cross-session PII를 operator UI 기본 payload로 노출하는 것 금지
- booth customer flow에 operator diagnostics나 recovery history entry를 노출하는 것 금지
