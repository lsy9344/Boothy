## Critical Risks Requiring Immediate Attention

### 1) REQ-001: Missing camera connect/health implementation

**Score: 9 (Critical)**  
**Probability**: High — current `get_camera_status` does not connect or observe camera state.  
**Impact**: High — Customer Mode readiness gating can permanently block (or provide false confidence) and cannot meet AC2–3.

**Mitigation (preventive)**:
- Implement an EDSDK-backed camera session lifecycle with reconnect + periodic health checks.
- Emit status updates via Tauri events so UI can reflect real state without polling.

**Testing Focus**:
- Windows manual smoke with real EOS 700D: connect/disconnect/reconnect, cable unplug, sleep/wake, error recovery.
- Regression: ensure UI remains responsive during background health checks.

## Risk-Based Testing Strategy

### Priority 1 (Critical)

- Tethering enabled (Windows + configured EDSDK path) → camera connect → UI shows Ready.
- Cable unplug / camera sleep → UI transitions to Not Ready with customer-safe message + admin diagnostics.
- Reconnect / retry path restores Ready without app restart.

### Priority 2 (High)

- Customer Mode gating (Setup + Capture) blocks progression only when tethering is required and camera is not ready.
- Rollback semantics: disabling tethering (or clearing EDSDK path) removes gating and allows file-based/manual mode.
