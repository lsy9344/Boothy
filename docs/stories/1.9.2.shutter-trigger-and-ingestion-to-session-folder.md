# Story 1.9.2: Shutter Trigger + Ingestion to Session Folder

## Status
Ready for Review

## Story
**As a** studio customer,  
**I want** the “Shoot” button to trigger the camera and automatically ingest the captured file into my session folder,  
**so that** I see the latest photo immediately without manual importing or file browsing.

Scope: Implement EDSDK shutter trigger from Customer Mode and ingest/download captured files into the active session folder, with idempotency against duplicate/partial events. Emit a UI-friendly event when ingestion is complete so the latest photo can be auto-selected. Keep kiosk safety boundaries: write only to the active session folder and do not expose filesystem paths in Customer UI.

Out of scope: Camera connect/disconnect/reconnect + health monitoring (Story 1.9.1), sidecar auto-create / capture preset application (Story 1.9.3), Live View (Story 1.10).

Assumptions / Clarifications:
- Story 1.9.1 provides a running tethering controller with an open EDSDK session + registered object event callbacks.
- Story 1.9.4 provides physical connect/disconnect signal via Windows `WM_DEVICECHANGE` so "ready" gating does not require EDSDK polling for presence.
- For this story, ingestion completion is surfaced via a Tauri event named `image-added` (reusing the existing Customer Mode listener). Payload should be the ingested file path string; Customer UI must never render this path.
  - Note: Epic 1.9 previously referenced `image-ingested`; we standardize on `image-added` to match the existing listener in `upstream/RapidRAW/src/modes/screens/CustomerModeSetup.tsx`.

## Acceptance Criteria
1. Customer Mode “Shoot” triggers a Windows-only capture via EDSDK when tethering is enabled and camera status is `ready` (Story 1.9.1 / `get_camera_status`).  
2. The captured file is downloaded into the active session folder (`get_active_session_metadata().sessionFolderPath`) and becomes visible in the session gallery; the newest image is auto-selected within 2 seconds (NFR1).  
3. The backend capture command enforces guards (defense-in-depth): active session exists, not in ExportLock, tethering enabled, camera ready; failures return a customer-safe error (no raw SDK codes/paths in Customer UI).  
4. Duplicate/partial EDSDK object events do not create duplicate files or double-enqueue downstream processing (idempotent behavior). Incomplete transfers are cancelled/cleaned and never emit `image-added`.  
5. On successful ingestion completion, backend emits Tauri event `image-added` so the UI can refresh without polling (Customer UI must not render the path).  
6. Non-Windows builds remain unblocked: the command compiles and returns “not supported” without side effects.

## Dev Notes

### Data / Contracts
- Destination folder is always the active session folder from session metadata (FR7).
- Ingestion completion event (this story): `image-added` (payload: ingested file path string). Customer UI must not render the path; it is used only for internal refresh + downstream integration.
- UI update should prefer event-driven refresh (`image-added`) instead of polling folder contents.
- Error contract: `tether_take_picture` returns a customer-safe error string for Customer Mode display (no filesystem paths, no raw SDK codes). On non-Windows, return a stable “not supported” error string without side effects.

### Key Integration Points
- Backend tethering module: `upstream/RapidRAW/src-tauri/src/tethering/*` (FFI + controller)
  [Source: docs/architecture/source-tree.md#boothy-enhancement-code-placement-권장]
- Active session folder source of truth: `upstream/RapidRAW/src-tauri/src/session_management.rs#get_active_session_metadata`
  [Source: docs/stories/1.3.reservation-check-session-folder-creation.md]
- Customer UI capture: `upstream/RapidRAW/src/modes/screens/CustomerModeCapture.tsx`
- Customer UI gallery refresh: `upstream/RapidRAW/src/modes/screens/CustomerModeSetup.tsx` (list + auto-select newest)

### EDSDK Notes (Capture + Transfer)
- Trigger capture using `EdsSendCommand(..., kEdsCameraCommand_TakePicture, ...)`.
- Expect transfer notifications via object events (typically `kEdsObjectEvent_DirItemRequestTransfer`) and download with `EdsDownload` → `EdsDownloadComplete` (or `EdsDownloadCancel` on failure).
  [Source: EDSDK_h/EDSDK.h; EDSDK_h/EDSDKTypes.h]
 
Idempotency key (implementation guidance):
- Prefer a key derived from the EDSDK directory item identity (e.g., the `EdsDirectoryItemRef` for the transfer event, or stable item info if available) so duplicate object events for the same item do not cause double-download or double-emit.

### Risks / Watch-outs
- EDSDK can emit multiple object events per capture; design dedupe keys carefully.
- Downloads must be atomic enough to avoid partial file reads by preview/export pipelines.

## Testing
- Unit: dedupe/idempotency helpers; error handling paths (not-ready, disabled, ExportLock).
- Unit: ensure `image-added` is emitted only after full download completion (no `.part` leaks) and never twice for the same capture.
- Manual (Windows + EOS 700D): shoot repeatedly; simulate cable disconnect mid-transfer; confirm safe error + no UI freeze; confirm newest image auto-selected within 2s.
