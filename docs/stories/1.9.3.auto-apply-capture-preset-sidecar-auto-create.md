# Story 1.9.3: Auto-Apply Capture Preset (Sidecar Auto-Create for New Captures)

## Status
Ready for Review

## Story
**As a** studio customer,  
**I want** newly captured photos to automatically use my selected “capture preset”,  
**so that** the preview (and later export) reflects my chosen look without extra steps.

Scope: Provide a Customer Mode capture preset selection that is persisted for the active session and applied to **newly captured images only** by auto-creating a sidecar when missing. Ensure preset changes only affect subsequent captures and never rewrite existing sidecars. Keep compatibility with RapidRAW sidecar/preset concepts.

Out of scope: Camera connect/disconnect/reconnect + health monitoring (Story 1.9.1), shutter trigger + file ingestion (Story 1.9.2), Live View (Story 1.10), camera-setting lock/guardrails (FR16 / Story 1.12+).

## Acceptance Criteria
1. Customer Mode `Setup` provides a capture preset selection UI using the existing preset library, and the selected preset is persisted in the active session context.  
2. When a new image is ingested, if its sidecar does not exist, a `.rrdata` sidecar is created initialized with the selected capture preset’s adjustments.  
3. Preset changes apply only to subsequent captures; previous captures are not modified (no retroactive sidecar rewrites).  
4. Customer UI shows “instant feedback”: the most recent captured photo appears with the applied preset; if processing takes time, show a non-blocking placeholder until ready.  
5. Existing sidecar-based edit/restore behavior remains compatible (no regression in Admin Mode workflows).

## Dev Notes

### Definitions (MVP)
- “Capture preset” is stored as `SessionMetadata.capturePresetId` and resolves to a RapidRAW `Preset.id` from `presets.json` (loaded via existing preset infrastructure).  
- “None”/no-op is represented by `capturePresetId = null` and results in empty/no-op adjustments for sidecar auto-create.
  - Note: “leaf presets only (folders flattened)” is an MVP product decision for Customer Mode; confirm this policy is acceptable.

### Data / Contracts
- “Selected capture preset” must be available at the moment a new image is ingested; use session metadata as the single source of truth so backend can apply without depending on frontend state.  
- Sidecar format must match RapidRAW `.rrdata` schema (JSON `ImageMetadata` as written by `save_metadata_and_update_thumbnail`); do not invent a new sidecar format.  
- Sidecar creation must not leak filesystem paths into Customer UI.
- Ingestion completion event is `image-added` (Story 1.9.2). It must be emitted only after the sidecar auto-create step completes (if sidecar was missing), so the latest preview reflects the preset immediately.

### Relevant Source Tree / Integration Points
- Frontend presets: `upstream/RapidRAW/src/hooks/usePresets.ts`  
- Customer Setup UI: `upstream/RapidRAW/src/modes/screens/CustomerModeSetup.tsx`  
- Customer Capture UI: `upstream/RapidRAW/src/modes/screens/CustomerModeCapture.tsx`  
- Session metadata (FE): `upstream/RapidRAW/src/types/sessionTypes.ts`, `upstream/RapidRAW/src/modes/customerModeTypes.ts`  
- Session metadata (BE): `upstream/RapidRAW/src-tauri/src/session_management.rs`  
- Presets storage (BE): `upstream/RapidRAW/src-tauri/src/file_management.rs#load_presets` (`app_data_dir/presets/presets.json`)  
- Sidecar read/write + `.rrdata` naming: `upstream/RapidRAW/src-tauri/src/file_management.rs`

### Risks / Watch-outs
- Sidecar schema/version compatibility: ensure generated `.rrdata` is valid for existing restore/edit paths.
- Race conditions: ingestion may complete before preset selection is persisted; define a safe default behavior.

## Testing
- Unit: sidecar creation only when missing; preset-change semantics (only subsequent captures); atomic write helper; “preset not found” fallback to no-op adjustments.
- Manual: verify Admin Mode edits/exports still restore from existing sidecars; verify capture flow meets “latest shot visible within 2s” target (NFR1).

### Reviewed By: Quinn (Test Architect)

### Code Quality Assessment

The capture preset selection UI and session persistence are implemented cleanly, and the backend sidecar helper uses an atomic write. However, core behavior is not wired into the capture ingestion pipeline: `image-added` still emits before sidecar creation and the auto-create helper is only invoked in `CustomerModeSetup`, which is unmounted during actual capture. This leaves AC2/AC4 unmet and introduces a race where a later preset change could affect an earlier capture. Prerequisites are also not met: story status is not "Review", tasks are unchecked, and validation gates were not run.

### Requirements Traceability

- AC1 (preset selection UI + persisted session): Given presets are loaded in Setup, when a customer selects a preset, then the selection is persisted via `upsert_session_metadata` and session context updates. Validation: code review only (no automated test).
- AC2 (auto-create sidecar on ingestion): Given a new capture without a sidecar, when ingestion completes, then a `.rrdata` sidecar is created with the selected preset before `image-added`. Validation: not met; creation happens after `image-added` and only in Setup UI.
- AC3 (preset changes apply only to subsequent captures): Given a preset change after a capture, when the prior capture is processed, then its sidecar should not use the new preset. Validation: at risk; sidecar creation uses current session metadata at time of invocation.
- AC4 (instant feedback with preset-applied preview + placeholder): Given a capture in progress, when processing completes, then the latest photo preview appears with the preset, with a non-blocking placeholder while pending. Validation: not met; Capture screen has processing copy but no preview, and Setup grid placeholders only render when Setup is mounted.
- AC5 (Admin sidecar behavior compatibility): Given Admin workflows, when editing/restoring, then sidecar behavior remains unchanged. Validation: code review only.
