# Story 1.9.1: Tethering Controller (Session Lifecycle + Status Events)

## Status
Ready for Review

## Story
**As a** studio operator,  
**I want** the kiosk to maintain a real EOS 700D camera session with event-driven status updates,  
**so that** Customer Mode can safely gate capture and Admin can recover from disconnects without restarting the app.

Scope: Turn Story 1.8 tethering scaffolding into a real backend controller: EDSDK init/terminate, camera open/close session, event handler registration/unregistration, and basic health monitoring. Wire Admin Diagnostics Connect/Disconnect/Retry to real commands and emit camera status changes as Tauri events so UIs update without manual refresh.

Out of scope: Shutter trigger + file ingestion (Story 1.9.2), sidecar auto-create / capture preset application (Story 1.9.3), Live View (Story 1.10), camera-setting lock/guardrails (FR16 / Story 1.12+).

## Acceptance Criteria
Traceability: Epic Story 1.9 AC6 split; aligned to PRD FR8/FR22 and NFR4.  
1. Backend has a single â€śtethering controllerâ€ť that owns EDSDK lifecycle and camera session lifecycle (init/terminate, open/close session, register/unregister handlers).  
2. Admin Diagnostics `Connect / Disconnect / Retry` actions invoke real backend commands and update UI state based on results.  
3. Camera status changes are pushed to UI via a Tauri event (e.g., `camera-status-changed`) and both Customer/Admin UIs react without manual refresh.  
4. Basic health monitoring exists to move to a safe status on disconnect/sleep/error and provides a clear retry path. (Customer: â€śGet Helpâ€ť; Admin: details + retry)  
5. App shutdown and privacy reset cleanly terminate the controller and release all EDSDK resources (no deadlocks/crashes).  
6. Non-Windows builds remain unblocked (stubs/guards compile and return â€śnot supportedâ€ť).
