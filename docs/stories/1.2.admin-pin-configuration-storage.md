# Story 1.2: Admin PIN & Configuration Storage

## Status
Done

## Story
**As a** studio operator,  
**I want** to configure kiosk settings (PIN, session root, export rules, EDSDK path) in one place,  
**so that** the kiosk can be operated consistently without ad-hoc manual steps.

## Acceptance Criteria
1. Admin PIN 설정/변경이 가능하며, 저장 시 평문이 아니라 해시 기반으로 저장된다.
2. 설정 항목을 제공한다: 세션 루트 폴더, 세션 폴더명 템플릿, Export 출력 경로/품질/리사이즈, 파일명 템플릿, UI Lock-down 레벨(0~2), EDSDK 경로.
3. 설정은 Tauri `app_data_dir`에 영속 저장되며 재실행 후에도 유지된다.
4. 필수 설정(경로/권한 등)이 유효하지 않으면 Customer Mode는 “진입 불가 + 도움 요청”으로 안전하게 차단된다.
5. 설정 초기화/기본값 복원이 가능하다(운영 롤백).

## Dev Notes

### Goal / Context
- This story establishes a single, persistent kiosk configuration surface used by subsequent Customer Mode stories (session creation, reservation check, export rules, tethering) and hardens customer safety by gating Customer Mode entry when required config is invalid.
  [Source: docs/prd/epic-1-rapidtetherraw-customer-mode-mvp.md#epic-1-rapidtetherraw-customer-mode-mvp]

### Dependencies
- Story 1.1 (AppShell + Customer/Admin routing + Admin entry UX) must be implemented before starting Story 1.2 development work.
  [Source: docs/stories/1.1.mode-shell-guided-state-machine-scaffold.md#tasks--subtasks]

### Previous Story Insights
- Story 1.1 implemented a minimal Admin escape hatch with a temporary PIN (`0000`) and a Customer Mode enable flag stored in `localStorage` (`boothy.customerModeEnabled`).
  [Source: docs/stories/1.1.mode-shell-guided-state-machine-scaffold.md#tasks--subtasks]
- Backend `cargo check` was blocked in this repo due to missing upstream `rawler` submodule; Story 1.2 includes backend work so ensure upstream dependencies are initialized before relying on Rust validation.
  [Source: .ai/debug-log.md#2026-01-02---backend-cargo-check-limitation]

### Tech Stack (Relevant)
- Frontend: React 19 + TypeScript (Vite), Tailwind, Tauri 2
  [Source: docs/architecture/tech-stack.md#frontend]
- Backend: Rust (edition 2024), Tauri 2.9; config/presets commonly use `serde/serde_json`
  [Source: docs/architecture/tech-stack.md#backend]

### Configuration & Persistence Guidance
- Persist kiosk configuration under Tauri `app_data_dir` (PIN, session rules, export rules, lock-down level, EDSDK path); do not store PIN in plaintext.
  [Source: docs/prd/technical-constraints-and-integration-requirements.md#deployment-and-operations]
- Prefer starting with a JSON store for MVP and keep a storage interface boundary for a future SQLite migration.
  [Source: docs/prd/technical-constraints-and-integration-requirements.md#integration-approach]
  - Lock-down level(0~2) is persisted/configured in this story; enforcement details are handled in later ops/guardrails work (Story 1.12).
    [Source: docs/prd/epic-1-rapidtetherraw-customer-mode-mvp.md (Story 1.12)]

### Source Tree / Integration Points
- Frontend entrypoints: `upstream/RapidRAW/src/main.tsx` and the existing App shell structure; keep invoke/event contract changes minimal.
  [Source: docs/architecture/source-tree.md#upstream-rapidraw-핵심-엔트리포인트]
- Backend entrypoints: `upstream/RapidRAW/src-tauri/src/main.rs` (commands), and `file_management.rs` (existing settings/preset/sidecar-related utilities).
  [Source: docs/architecture/source-tree.md#upstream-rapidraw-핵심-엔트리포인트]

### Customer Safety & Error UX
- Treat invalid config (paths/permissions/SDK prerequisites) as a safe “entry blocked” state for customers, with Admin-only diagnostics and remediation.
  [Source: docs/architecture/coding-standards.md#backend-rusttauri]
- Customer-facing errors should remain “short + specific + one help action”.
  [Source: docs/architecture/coding-standards.md#frontend-reacttypescript]

### Scope Notes (EDSDK / Platform)
- EDSDK is Windows-only; validation and any SDK-specific checks must not block macOS development flows (use platform guards / feature flags).
  [Source: docs/architecture/coding-standards.md#backend-rusttauri]

### Testing & Validation Expectations
- Minimum gates: frontend build + backend `cargo check` (and/or `cargo test` when feasible) + smoke scenarios for config, PIN, and gating.
  [Source: docs/architecture/coding-standards.md#testing-validation-권장-최소-게이트]

## Testing

- Integration: config read/write persistence across restart; validation gating blocks Customer Mode when required settings invalid.
- Manual smoke: Admin sets PIN/config → restart → verify Admin login + config persistence; reset restores defaults; invalid paths produce “도움 요청” only.
