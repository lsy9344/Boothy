# Story 1.9.4: WM_DEVICECHANGE Physical Camera Connection Detection (Windows)

## Status
Ready for Review

## Story
**As a** studio operator,  
**I want** camera connect/disconnect status to be detected via Windows device notifications (`WM_DEVICECHANGE`) instead of EDSDK polling,  
**so that** cable unplug/replug and device-level events are reflected immediately and reliably in Customer/Admin UI.

Scope:
- Implement Windows-only "physical connection watcher" using `WM_DEVICECHANGE`
- Use the watcher to drive `CameraStatus` transitions and the `camera-status-changed` Tauri event
- Ensure `get_camera_status` returns controller/watch state and does **not** call EDSDK APIs just to determine whether a camera is connected

Out of scope:
- Shutter trigger + ingestion (Story 1.9.2)
- Live view (Story 1.10)
- Camera-setting lock/guardrails (FR16 / Story 1.12+)

## Acceptance Criteria
1. Windows: When the EOS 700D USB cable is plugged/unplugged while the app is running, backend emits `camera-status-changed` and UI updates without manual refresh within 1 second.
2. Windows: When the app starts and the camera is already connected, initial camera status reflects "connected" correctly (startup enumeration; no "wait for first event" bug).
3. `get_camera_status` does not call into EDSDK (no dynamic loading/SDK init) to infer camera presence; it returns the last known state from the watcher/controller.
4. Device filtering exists so that unrelated USB device changes do not change camera status (minimum: Canon EOS device class + VID/PID or HardwareId match; exact matching rule documented).
5. On device removal, tethering controller transitions to a safe state and releases session resources without deadlocks/crashes.
6. Physical connection status is detectable even when EDSDK is not configured; if the camera is physically connected but EDSDK path is missing/invalid, status becomes `connected_not_ready` with an Admin-actionable message (Customer-safe summary only).
7. Non-Windows builds remain unblocked (feature guards/stubs compile and return `not supported`).

## Dev Notes

### WM_DEVICECHANGE reference (Windows)
- Listen for `WM_DEVICECHANGE`
- `wParam` commonly used:
  - `DBT_DEVICEARRIVAL`
  - `DBT_DEVICEREMOVECOMPLETE`
  - `DBT_DEVNODES_CHANGED` (useful as a "re-enumerate" signal)
- `lParam` points to `DEV_BROADCAST_HDR` / `DEV_BROADCAST_DEVICEINTERFACE` when `dbch_devicetype == DBT_DEVTYP_DEVICEINTERFACE`

### Recommended design constraints
- "Physical connected" should not require EDSDK installation/path configuration.
- Status updates should be event-driven; avoid tight polling loops.
- When device removal happens mid-operation, prefer safe shutdown + clear retry path over best-effort continuation.

### Recommended filtering rule (practical)
- Prefer a config-driven allowlist of device identifiers (substring match), e.g. Device Manager -> device properties -> Hardware Ids (contains `VID_04A9` for Canon).
- If only one camera model is supported, start with a strict allowlist to avoid noise from unrelated USB devices.
